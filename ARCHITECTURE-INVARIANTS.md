# Architecture Invariants — Folio

> Folio's confidence comes from a small set of **convergence points** — single places where each cross-cutting property is decided. This doc names them so `/code-review` and `/shakeout` can FLAG any path that bypasses one, instead of re-auditing wiring/dedup/safety every session. Authored 2026-06-02 by `/architecture-invariants audit`, with each convergence point verified by grep against source (consumer counts noted). Sibling to the per-plan `## Threat model` sections (which do this for attacks).

**How to read an invariant:** each names a CONVERGENCE POINT (the one place a property is decided, with `file:symbol`) and the BYPASS that is a bug (a path that does the property itself instead of routing through the convergence point). Reviews check diffs against these, mechanically. **A bug is almost always a path that skips a convergence point** — the one CRITICAL finding of the 2026-06-01 auth audit (a member minting a token above their role) was exactly that: the single write path that bypassed invariant 7.

## Invariants (converged — enforced)

1. **Authentication identity: every request's identity is the `AuthContext` primitive set by the auth middleware — nothing re-derives identity another way.** Converges on `apps/server/src/middleware/auth.ts` (`AuthContext`, `attachUser`) + `middleware/bearer.ts` (`attachToken`) — the ONLY setters of `c.set('token' | 'user' | 'authMethod')`. Every downstream guard reads `c.get('token')`, never the raw `Authorization` header or cookie. A handler that parses the raw credential a second time (or trusts a caller-supplied identity) is a bug — read the context primitive. *(This closed the "stray Authorization header downgrades a session" class — B round 3 fix #1.)*

2. **Authorization (tools): every agent tool call passes the `executeTool` scope double-check — token scope ∩ caller scope, fail-closed.** Converges on `apps/server/src/lib/agent-tools.ts:282` (`executeTool` — its own docstring: "the ONE dispatch+auth point that both transport faces call"). Both the MCP route and the in-process runner call it; missing `callerScopes` defaults to `[]` (deny), never wildcard. A tool path that checks scopes itself, or runs a tool outside `executeTool`, is a bug — route it through the dispatcher.

3. **Authorization (project ceiling): an actor's effective project reach is `agent ∩ token ∩ caller`, computed in one place.** Converges on `apps/server/src/lib/agent-projects.ts` (`intersectAgentProjects`, `resolveAgentProjects`, `callerProjectsFor`), folded into the run token once at `runner.ts:~336` and enforced by `requireResource()` (`middleware/bearer.ts:140`). 10 route files + the runner + SSE all read this same intersect. A path that widens projects, or re-derives the ceiling its own way, is a bug — use the shared intersect.

4. **HTTP authorization: mutating routes are gated by `requireScope` (+ `requireResource` for project-scoped), and session-only routes reject token callers via `requireSessionUser`.** Converges on `middleware/bearer.ts` (`requireScope`, `requireResource`, `requireUserOrToken`) + `middleware/auth.ts` (`requireSessionUser`). A mutation route mounted without a scope guard, or an auth-grant/credential route reachable by a bearer token, is a bug.

4a. **Visibility: "what may this human actor SEE?" is decided ONLY by `lib/access.ts` — never by an inline grant/role query.** Converges on `apps/server/src/lib/access.ts`: `userRole` (instance role from `users.role`), `canSeeWorkspace` (owner | `workspace_access` | a `project_access` TRAVERSE into the ws), `canManageWorkspace` (owner | `workspace_access`, NO traverse), `canSeeProject`, and `visibleProjectIds` (the set used by `/events` SSE narrowing, the ws-scoped runs list, and the project list). This replaced the dropped `memberships`-based visibility (Phase 4, drop-workspace-tenancy: one instance = one team; workspaces are org folders, not a security boundary). A route/SSE/list that hand-rolls `eq(workspaceAccess…)` / `eq(projectAccess…)` / `users.role` to decide visibility — instead of calling these helpers — is a bug (it's how the CR-7..CR-11 cross-project leaks happened: the narrowing was wired into one surface and not the convergence point). Empty `visibleProjectIds` set = DENY, never allow-all. *(Ratified 2026-06-05 in Phase 4 — the single who-can-see-what decision; the structural twin of invariant 3's project ceiling for the human-actor side.)*

5. **Data access (server): every write goes through `txWithEvents` so the mutation and its event are one atomic unit — and the event's scoping labels (`workspaceId`/`projectId`) are the SAME ones the mutation was authorized against, never a second caller-supplied value.** Converges on `apps/server/src/lib/events.ts` (`txWithEvents`, `emitEvent`) — 16 server files. This IS the architectural "every write emits an event" rule (CLAUDE.md). A write that inserts/updates outside `txWithEvents`, or emits no event, is a bug — wrap it. **A second, subtler bug:** a handler that authorizes a mutation against project P but then emits the event with a *different* `projectId` (a free typed value, a stale variable, `null`) — the SSE subscription filter (`routes/events.ts`) narrows by that label, so a mislabeled event is durably stored and replayed under the wrong scope. The label passed to `emitEvent` MUST be the resource's authorized scope, derived from the same `requireResource`/`canSeeProject` decision that let the write through — not re-supplied independently. *(Sharpened 2026-06-06 after the SSE-authorization audit: subscription-time filtering blocks the *leak* but not the *mislabel*; the convergence point owns atomicity AND label-fidelity. Residual is LOW — filtering catches downstream — but the check belongs at every `emitEvent` call site, not only at subscription.)*

6. **Data access (web): all HTTP goes through the one `client`, and react-query keys come from the per-resource key factories.** Converges on `apps/web/src/lib/api/client.ts:48` (`client`, ~25 importers) + 19 `*Keys` factory objects. A bare `fetch` outside the client, or an ad-hoc string query-key that should use a factory, is a bug — drift risk on invalidation. *(Known soft spots: 3 sites hand-build literal keys that mirror a factory — `documents.ts:190`, `projects.ts:78`, `routes/w.$wslug.tsx:301` — tracked in `tasks/retro-follow-ups.md`.)*

7. **Token authority never exceeds the minting caller's role.** Converges on `roleToScopes` (`apps/server/src/lib/agent-schema.ts:118`) — the role→scope ceiling. A write path that accepts caller-supplied scopes (or any authority) without intersecting `roleToScopes(role)` is a bug — route it through the ceiling. *(This is the invariant whose bypass at `routes/tokens.ts` was the 2026-06-01 CRITICAL privilege-escalation fix `9f75c40`.)*

8. **Live updates: SSE teaches react-query WHEN to refetch; the event stream is not a source of truth EXCEPT for the two ratified live-tail/broadcast consumers below.** Converges on `apps/web/src/lib/api/event-stream.ts:49` (`useEventStream` — docstring: "owns NO state and is NOT a source of truth"), 7 consumers. The default: events invalidate queries; the canonical GET refetches. A NEW consumer that builds UI state directly from an event payload (instead of invalidate-and-refetch) is a bug — it desyncs — UNLESS it is a broadcast-only datum with no GET, or an append-only live-tail feed, in which case it must be added to Deliberate exceptions (two such consumers are ratified there today). SSE filters match by **id**, cache keys by **slug** — also see Deliberate exceptions.

9. **Error handling: failures throw `HTTPError` → the `{ error: { code, message } }` envelope → the client surfaces via `formatApiError`.** Converges on `apps/server/src/lib/http.ts:25` (envelope) + `apps/web/src/lib/api/errors.ts:23` (`formatApiError`, 27 web files). A handler that returns a bespoke error shape, or swallows an error (`catch {}` discarding it), or a UI that hand-formats an error instead of `formatApiError`, is a bug.

10. **Entity modeling: new entity types and fields are DATA (frontmatter) before they are tables.** Converges on the `documents` schema (`apps/server/src/db/schema.ts:232-237`): 3 typed columns (`title`, `status`, `body`) + a `frontmatter` JSON blob; "everything else lives in frontmatter for maximum flexibility." Agents, triggers, and (Phase 3) skills/memory are `type:'agent'`/`'page'` documents, NOT new tables. A migration adding a column for what is clearly an attribute of one type — or a new table for what could be a document type — is a bug. Re-confirm before adding any `*.sql` that adds a column.

11. **Skill trust: the `trusted` flag is a TYPED COLUMN on `instance_skills`, set ONLY via `setSkillTrust` (gated by `canBlessSkill`).** Converges on `apps/server/src/lib/skill-trust.ts` (`canBlessSkill` — bless iff session user OR system-origin operator `createdBy IS NULL`; `setSkillTrust` — the one sanctioned mutator, which flips the `instance_skills.trusted` column). `trusted` is load-bearing: a blessed skill loads as TRUSTED INSTRUCTIONS into agent system prompts (`runner.ts` `buildSkillsPreamble`), an unblessed one rides the untrusted DATA envelope. **The forging path is closed STRUCTURALLY (Phase 4, drop-workspace-tenancy): because `trusted` is its own column — NOT a frontmatter key — a body/frontmatter write (import, restore, clone, edit, a raw `.update(instanceSkills).set({frontmatter})`) physically CANNOT reach it.** This supersedes the prior `__system`-skills-page model where `trusted` rode frontmatter and had to be defensively stripped on every write (`stripManagedSkillTrust`/`isSystemSkillPage` — both DELETED in Phase 4). The seeder (`instance-skills.ts`) strips any `trusted` key out of stored frontmatter and sets the column directly. The only way to bypass would be a NEW write surface that touches the `trusted` *column* outside `setSkillTrust` — there is none today. Additionally, `set_skill_trust` carries `unattendedFloor: true` so trust-elevation is REFUSED on an unattended (trigger-fired) run at the `executeTool` convergence point (`agent-tools.ts` — `UNATTENDED_FLOORED_SCOPES` OR `def.unattendedFloor`): the operator token is `createdBy`-null so `canBlessSkill` would otherwise pass even on a no-human run over attacker-supplied content. *(Ratified 2026-06-03 on the Piece-B diff; re-pointed to the typed `instance_skills.trusted` column 2026-06-05 in Phase 4 — the structural guarantee REPLACES the strip-on-write defense; the `skill.trust.changed` event was dropped with the `__system` teardown.)*

12. **Irreversible-op confirmation is a property of the OPERATION (its risk tier), not of the caller's SURFACE.** An irreversible/high-risk tool call must be gated by an explicit human confirmation step BEFORE it executes, and which calls need that gate is decided in ONE place by the operation's risk tier — never inferred from "which entry point is this." Converges on `apps/server/src/lib/agent-tools.ts` (`effectiveRiskTier(def)` — write/delete scopes default to `high`; the `pending_ops` record + `choice_card` emit + confirm-and-replay-recorded-params flow at `executeTool`'s Task-7 gate). A tool whose risk tier should require confirmation but executes without one — OR a new high-risk tool that defaults to `low` because nobody set its tier — is a bug: tier the operation, don't trust the surface. **KNOWN GAP — the gate is currently surface-coupled, which is the thing this invariant exists to drive OUT.** The confirm flow engages only when `caller.conversationId` is set (the operator cockpit chat); a **headless MCP `tools/call`** carries no `conversationId`, so the gate is SKIPPED and a high-risk op falls back to scope-only authority. Today's defense for the headless path is therefore invariant 2 (scope) + invariant 7 (role ceiling) ALONE — a leaked MCP token with `agents:write`/`config:write` gets unattended high-risk writes with no confirmation. This is documented-acceptable for trusted admin automation TODAY, but it is a surface deciding a property the operation should own. The convergence target is: make `effectiveRiskTier` the sole decider on EVERY path (conversation, headless MCP, trigger-fired), with the *response* to a needed-but-impossible confirmation (no human present) being a fail-closed REFUSAL — the way `unattendedFloor` already refuses trust-elevation on a no-human run (invariant 11) — rather than a silent fall-through to scope-only. *(Named 2026-06-06 after the MCP-authorization audit. Listed under enforced invariants because the convergence point — `effectiveRiskTier` at `executeTool` — EXISTS and is correct; the gap is that one surface bypasses it. Track the close in `tasks/retro-follow-ups.md`; until then, the headless bypass is a Deliberate exception below.)*

13. **Agent-doc resolution from a token: `token.agentId → agent Document` is decided in ONE place, which resolves the operator's synthetic sentinel.** Converges on `apps/server/src/lib/agent-tools-registry.ts` (`resolveAgentDocForToken`) — used by all 7 registry resolution sites (project-resolve, comment-author, describe_workspace, list_projects, find_documents, get_agent_self, the run-tools allow-list). The operator is a CODE SINGLETON: its token carries `OPERATOR_AGENT_ID` (`operator:_operator`), which has NO `documents` row, so a raw `findFirst({id})` MISSES and throws `agent_missing` (or silently degrades to `['*']`). The helper resolves the sentinel to `getOperatorDocument()`; any other agentId still throws `agent_missing` if absent (real guard intact). The check is on the sentinel VALUE (server-set at mint, un-forgeable — a real agent id is a UUID; the value can't be stored in a token's `agent_id` FK). A new site that open-codes `findFirst({id: token.agentId, type:'agent'})` is a bug — route it through the helper, or it will mis-handle the operator. *(Named + completed 2026-06-06 in the cockpit shake-out — the bug it fixes (operator `agent_missing` on every project-scoped tool) was caused by this resolution being scattered. ONE local exception, cycle-bound: `lib/agent-guards.ts` cannot import the registry helper — registry imports agent-guards — so it has its own `resolveCallingAgent` with the same sentinel handling; keep the two in lockstep.)*

14. **Skill rendering into the prompt: the trusted/untrusted skill body + label is rendered in ONE place, shared by every agent execution path.** Converges on `apps/server/src/lib/skill-preamble.ts` (`renderTrustedSkills`/`renderUntrustedSkills` + `TRUSTED_SKILLS_LABEL`/`UNTRUSTED_SKILLS_LABEL`) — invariant 11's prompt-injection boundary depends on the wording matching across paths. The document path (`buildInitialMessages`), the conversation/cockpit path (`skillsToMessages`), and the cc path all render through these. A path that hand-rolls a skill label/body (bypassing the module) is a bug — the wording would drift and a hardening of the untrusted-DATA framing wouldn't reach it. *(Named 2026-06-06 — the absence of this convergence on the conversation path WAS the "operator doesn't follow its skill" bug; the cockpit path rendered no skill at all.)*

15. **FK-actor vs event-actor: a write's `created_by`/`updated_by` (FK → `users.id`) and its `events.actor` are resolved separately — the FK column gets a real user, the event actor keeps the agent slug.** Converges on `apps/server/src/lib/agent-tools-registry.ts` (`serviceActor(ctx)` — returns the FK-valid real user `ctx.confirmerId ?? token.createdBy`, fail-loud on a non-FK id) + the `eventActor` arg threaded through `create/update/deleteDocument`. For an agent/operator run `ctx.actor` is the slug `agent:<slug>` — NOT a `users.id` (writing it to an FK column is errno 787), but it MUST stay the event actor so the agent-chain autonomy gate (`isAgentOriginated` → `agent:` actor, invariant 7-adjacent) keeps firing. A write that puts `ctx.actor` in an FK column (787), or that lets the event actor become the human user for an agent write (defeats `FOLIO_AGENT_CHAINS_ENABLED`), is a bug. *(Named 2026-06-06 — the FK-787 on operator writes drove this split. Residual: the optional `eventActor` defaults to `actor.id`, so a future write service that forgets it silently re-collapses the two — tracked in `tasks/retro-follow-ups.md` as a structural follow-up.)*

## Open — NOT yet converged (gaps)

- **Live-update connection model:** `useEventStream` opens ONE `EventSource` per consumer (7 today), not a shared workspace stream. This is a deliberate v1 simplicity choice, NOT a missing convergence point — but it means "the live stream" is not yet a single connection. Tracked in `tasks/retro-follow-ups.md` (SSE fan-out, 2026-06-01) with an upgrade trigger (shared pool) if many panels coexist. Recorded here so it isn't mistaken for invariant 8 being violated.

- **MCP credential lifecycle:** MCP clients authenticate with a STATIC long-lived bearer token (`routes/mcp.ts` `requireToken`) — there is no OAuth-style / capability-handshake / short-lived-grant flow. This is NOT a gap in *enforcement* — the token's scopes route through the same `executeTool` double-check as every other path (invariant 2), so a stolen MCP token can do no more than its scopes allow. It is a gap in *credential lifecycle*: no rotation handshake, no per-session capability narrowing, no revocation-on-use. SAFE today because tokens are scoped + role-ceilinged (invariants 2/7) and minting is gated. WATCH-ITEM, not a fix-now: the MCP ecosystem is moving toward OAuth-style auth, and Folio's "agent is just another scoped API consumer" model is the right base to add it onto when MCP usage broadens beyond trusted first-party clients. Recorded here (2026-06-06, MCP-authorization audit) so the lifecycle question isn't forgotten — and so it's not mistaken for invariant 2 being unenforced (it is enforced).

*(All other audited properties are converged. No faked invariants.)*

## Deliberate exceptions (intentional bypasses — do not re-flag)

- **`workspace_access` / `project_access` / `instance_skills` are real relational TABLES, not `documents` rows (invariant 10)** — a justified exception to "entity types are DATA before tables." The two access tables are pure AUTHORIZATION data: a `(user, scope)` composite-PK join with a reverse index (`who can see this workspace?`), queried set-based on the hot path (every workspace/project-scoped request via `visibleProjectIds`/`canSeeWorkspace`). Modeling a grant as a `document` would make the convergence-point queries (joins, `IN`, existence checks) impossible. `instance_skills` is a typed table because its `trusted` flag MUST be a column, not frontmatter, to make trust-forging structurally impossible (invariant 11). These are correctly tables. (`apps/server/src/db/schema.ts` — `workspaceAccess`, `projectAccess`, `instanceSkills`.) *(Ratified 2026-06-05 in Phase 4 — recorded so the data-before-tables rule isn't re-flagged against them.)*
- **`useReactorHealth` writes the query cache directly (`setQueryData`) instead of invalidate-and-refetch (invariant 8)** — because reactor health is broadcast-only; there is NO GET endpoint, so the event IS the only source of truth for that one datum. (`apps/web/src/lib/api/provider-health.ts`.)
- **`useActivityFeed` builds live-tail rows directly from the SSE payload and lets "live win" over the history GET (invariant 8)** — because it is an append-only ACTIVITY LOG, not an editable record: a `agent.run.*` transition feed legitimately wants the event-derived row layered on top of the `useWorkspaceRuns` history seed. History comes from react-query; only the fresh live tail is event-derived. (`apps/web/src/lib/api/activity-feed.ts:24-28,77-78`.) *(Ratified 2026-06-02 after the invariant-auditor flagged it as a second, undocumented SSE-as-truth consumer — the doc's original "NEVER" over-claimed; invariant 8 now names the two allowed shapes explicitly.)*
- **SSE filters match by resource ID while cache keys are slug-based (invariant 8)** — this is correct, not a mismatch: the `/events` route matches `?project=`/`?parent=` against the event row's real id, while `documentsKeys`/`commentsKeys` key on slug. Two identifiers, two purposes. (Fix shipped `bb6da69`; documented at `use-live-documents.ts:13-16`.)
- **Human PATs skip `requireResource` (`bearer.ts:146`, `if (!token.agentId) return next()`) (invariant 3)** — deferred to a Phase 3+ per-PAT narrowing UI. SAFE only because invariant 7 now gates what scopes a PAT can hold; revisit when PAT project-narrowing ships. *(Open follow-up: a human PAT can still hold workspace-wide `config:write` un-narrowed.)*
- **`loadAgentDefinition` performs a SYSTEM-authority read that bypasses `executeTool` (invariant 2) and `requireResource` (invariant 4)** — the runner's ONE internal definitional-read exemption. It reads the agent's own body + the skills named in `frontmatter.skills` from the **`instance_skills` table, by name** (UPDATED 2026-06-05 in Phase 4 — was a `(__system, skills, page)` document read; skills are now instance-level rows, resolved by `getInstanceSkillsByNames`), to materialize the agent's capability (like its prompt) into the run's context. SAFE because it is NARROW and NOT REACHABLE: it matches `instance_skills` rows by EXACT frontmatter-named slug, throws `MISSING_SKILL` on any miss with NO broader fallback, and is module-private — NOT registered in `agent-tools-registry`, NOT a route, so a caller can never invoke it. It threads each skill's `trusted` column (invariant 11): only `trusted:true` skills enter the trusted system channel; unblessed skills ride the untrusted DATA envelope on both the API and cc paths. (`apps/server/src/lib/runner.ts` `loadAgentDefinition`, called only by `loadContext`.) *(Ratified 2026-06-02 (Phase B); trust-channel split 2026-06-03; re-pointed from the `__system` Skills project to `instance_skills` 2026-06-05 in Phase 4 — still NARROW + module-private, recorded so it isn't re-flagged.)*
- **The runner reads + decrypts the AI key by `(provider, ai_key_label)` with SYSTEM authority, bypassing `executeTool`/`requireResource` (invariants 2/4) and the per-workspace data path** — the runner's ONE internal credential read. AI keys are INSTANCE-level (`ai_keys` has no `workspace_id` since migration 0023): `loadContext` resolves the run's snapshotted `(provider, ai_key_label)` to a single `ai_keys` row, decrypts it, and injects the secret into the provider call ONLY — it never reaches a token, tool, response, the run messages, or frontmatter (threat model M1/M2/M6). UPDATED 2026-06-03 (instance-AI-config): this REPLACES the old B6 rule (key read by `run.workspaceId`); resolution now has NO workspace predicate, so a library/operator agent acting in any workspace resolves the shared instance key. SAFE because (a) the read is module-private in `loadContext` (NOT a tool, NOT a route — a caller can never invoke it), (b) the only WRITE surface for `ai_keys` is `/api/v1/instance/ai-keys`, session-only + `requireInstanceAdmin`, SECRET-classed so no agent token can reach it (invariant 4), and (c) usage is metered to the run's workspace (`ai_usage`) so the shared-key denial-of-wallet residual is attributable (per-key caps deferred). (`apps/server/src/lib/runner.ts` AI-key resolution in `loadContext`; route `apps/server/src/routes/instance-ai-keys.ts`.) *(Ratified 2026-06-03; the B6 key-resolution tests were INVERTED to this rule, not deleted.)*
- **The markdown-PATCH route does a raw `.update(documents).set({ frontmatter })` that bypasses the `updateDocument` service (invariant 5's write path) (`apps/server/src/routes/documents.ts` `isMarkdownRequest` branch)** — a wholesale frontmatter replace for `Content-Type: text/markdown|text/plain` writes. SAFE because it is still inside `txWithEvents`. *(Ratified 2026-06-03. The skill-trust concern that originally accompanied this exception is GONE as of Phase 4 (2026-06-05): skill trust moved off `documents` frontmatter onto the typed `instance_skills.trusted` column (invariant 11), so a `documents` frontmatter write — raw or not — physically cannot touch it; `stripManagedSkillTrust` was deleted.)*
- **Headless MCP `tools/call` skips the irreversible-op confirm gate (invariant 12)** — the Task-7 confirmation flow engages only with a `caller.conversationId` (the cockpit chat); a headless MCP call has none, so a high-risk op runs under scope-only authority (invariants 2 + 7) with no confirmation. This is the gap invariant 12 names as the thing to drive out; recorded here so the CURRENT state isn't re-flagged as a fresh bug each review while the close is pending in `tasks/retro-follow-ups.md`. ACCEPTABLE only for trusted first-party admin automation; revisit the moment a non-first-party MCP client (or a token that could leak to one) needs high-risk scopes. (`apps/server/src/lib/agent-tools.ts` Task-7 gate; `routes/mcp.ts` `tools/call`.) *(Recorded 2026-06-06, MCP-authorization audit.)*
- **Conversation/message/pending_ops writes bypass `txWithEvents` (invariant 5)** — chat persistence is walled off from the event stream BY DESIGN: emitting an event per chat turn would flood the SSE stream and fire the trigger-matcher on document-watching triggers. Conversations are not documents and not agent-reactable. Plain `db` transactions, no `emitEvent`. A test asserts creating a conversation + messages produces ZERO rows in `events` (threat model M10). (`apps/server/src/services/conversations.ts`.) *(Ratified 2026-06-05 in the operator cockpit chat work — recorded so the every-write-emits-an-event rule isn't re-flagged against the chat tables.)*
- **`conversations` / `messages` / `pending_ops` are new tables, not document types (invariant 10)** — a conversation must NEVER appear in `/documents` (no events, no trigger reach), needs purpose-built per-conversation `seq` ordering + streaming, and `pending_ops` is transient gate state for the irreversible-op confirm gate. Unlike `agent_run` (a document type behind a redacted `/runs`), there is NO read path that should surface a conversation as a document. (`apps/server/src/db/schema.ts` — `conversations`, `messages`, `pendingOps`.) *(Ratified 2026-06-05 in the operator cockpit chat work — recorded so the data-before-tables rule isn't re-flagged against them.)*

## How to use this doc

- **`/code-review` + `/shakeout`:** verify the diff against these invariants. FLAG (don't block) any path that bypasses a convergence point as a finding: "bypasses invariant <N> (<property>) — confirm intentional or route through <symbol>." The human accepts (and adds to Deliberate exceptions) or fixes. `/shakeout` auto-dispatches the `invariant-auditor` agent for this.
- **Plan-writers (`superpowers:writing-plans` + `netdust-core:architecture-invariants`):** a plan touching one of these properties cites the invariant in a `## Architecture invariants touched` note, so the implementer routes through the convergence point. (Phase 3 `folio_api` already does this — it rides invariants 2/3/4/7 via `app.request`, deliberately NOT hand-rolling auth.)
- **The built-in operator agent (Phase 3+):** reads this doc before editing Folio's own code/config. Staying inside the invariants is what makes the agent safe to let touch the app.
- **Keep it current:** add an invariant when a new convergence point is introduced; move a gap to Invariants when closed. A `/evaluate` retro that finds a bypass this doc didn't name should sharpen an invariant.
- **Traceability check (`bun run check:invariants`):** a cheap always-on layer that verifies each invariant's `Converges on` clause still POINTS AT REAL CODE — the cited `file.ts` exists, the named symbols are still defined there, and any embedded `:NN` still matches. It does NOT verify the property holds (that is the `invariant-auditor` agent's job via `/shakeout`); it only catches the doc drifting off its anchors. Errors (missing file/symbol) exit non-zero; line drift warns. The pre-commit hook (`scripts/hooks/pre-commit-invariants.sh`, installed by `scripts/hooks/install.sh`) runs it non-blocking when this doc or server/web source is staged. When you move a cited symbol, refresh the `:NN`; when you rename/remove one, fix the citation here.
