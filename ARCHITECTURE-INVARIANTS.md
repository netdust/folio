# Architecture Invariants тАФ Folio

> Folio's confidence comes from a small set of **convergence points** тАФ single places where each cross-cutting property is decided. This doc names them so `/code-review` and `/shakeout` can FLAG any path that bypasses one, instead of re-auditing wiring/dedup/safety every session. Authored 2026-06-02 by `/architecture-invariants audit`, with each convergence point verified by grep against source (consumer counts noted). Sibling to the per-plan `## Threat model` sections (which do this for attacks).

**How to read an invariant:** each names a CONVERGENCE POINT (the one place a property is decided, with `file:symbol`) and the BYPASS that is a bug (a path that does the property itself instead of routing through the convergence point). Reviews check diffs against these, mechanically. **A bug is almost always a path that skips a convergence point** тАФ the one CRITICAL finding of the 2026-06-01 auth audit (a member minting a token above their role) was exactly that: the single write path that bypassed invariant 7.

## Invariants (converged тАФ enforced)

1. **Authentication identity: every request's identity is the `AuthContext` primitive set by the auth middleware тАФ nothing re-derives identity another way.** Converges on `apps/server/src/middleware/auth.ts` (`AuthContext`, `attachUser`) + `middleware/bearer.ts` (`attachToken`) тАФ the ONLY setters of `c.set('token' | 'user' | 'authMethod')`. Every downstream guard reads `c.get('token')`, never the raw `Authorization` header or cookie. A handler that parses the raw credential a second time (or trusts a caller-supplied identity) is a bug тАФ read the context primitive. *(This closed the "stray Authorization header downgrades a session" class тАФ B round 3 fix #1.)*

2. **Authorization (tools): every agent tool call passes the `executeTool` scope double-check тАФ token scope тИй caller scope, fail-closed.** Converges on `apps/server/src/lib/agent-tools.ts:310` (`executeTool` тАФ its own docstring: "the ONE dispatch+auth point that both transport faces call"). Both the MCP route and the in-process runner call it; missing `callerScopes` defaults to `[]` (deny), never wildcard. A tool path that checks scopes itself, or runs a tool outside `executeTool`, is a bug тАФ route it through the dispatcher.

3. **Authorization (project ceiling): an actor's effective project reach is `agent тИй token тИй caller`, computed in one place.** Converges on `apps/server/src/lib/agent-projects.ts` (`intersectAgentProjects`, `resolveAgentProjects`, `callerProjectsFor`), folded into the run token once at `runner.ts:~336` and enforced by `requireResource()` (`middleware/bearer.ts:140`). 10 route files + the runner + SSE all read this same intersect. A path that widens projects, or re-derives the ceiling its own way, is a bug тАФ use the shared intersect.

4. **HTTP authorization: mutating routes are gated by `requireScope` (+ `requireResource` for project-scoped), and session-only routes reject token callers via `requireSessionUser`.** Converges on `middleware/bearer.ts` (`requireScope`, `requireResource`, `requireUserOrToken`) + `middleware/auth.ts` (`requireSessionUser`). A mutation route mounted without a scope guard, or an auth-grant/credential route reachable by a bearer token, is a bug.

4a. **Visibility: "what may this human actor SEE?" is decided ONLY by `lib/access.ts` тАФ never by an inline grant/role query.** Converges on `apps/server/src/lib/access.ts`: `userRole` (instance role from `users.role`), `canSeeWorkspace` (owner | `workspace_access` | a `project_access` TRAVERSE into the ws), `canManageWorkspace` (owner | `workspace_access`, NO traverse), `canSeeProject`, and `visibleProjectIds` (the set used by `/events` SSE narrowing, the ws-scoped runs list, and the project list). This replaced the dropped `memberships`-based visibility (Phase 4, drop-workspace-tenancy: one instance = one team; workspaces are org folders, not a security boundary). A route/SSE/list that hand-rolls `eq(workspaceAccessтАж)` / `eq(projectAccessтАж)` / `users.role` to decide visibility тАФ instead of calling these helpers тАФ is a bug (it's how the CR-7..CR-11 cross-project leaks happened: the narrowing was wired into one surface and not the convergence point). Empty `visibleProjectIds` set = DENY, never allow-all. *(Ratified 2026-06-05 in Phase 4 тАФ the single who-can-see-what decision; the structural twin of invariant 3's project ceiling for the human-actor side.)*

5. **Data access (server): every write goes through `txWithEvents` so the mutation and its event are one atomic unit тАФ and the event's scoping labels (`workspaceId`/`projectId`) are the SAME ones the mutation was authorized against, never a second caller-supplied value.** Converges on `apps/server/src/lib/events.ts` (`txWithEvents`, `emitEvent`) тАФ 16 server files. This IS the architectural "every write emits an event" rule (CLAUDE.md). A write that inserts/updates outside `txWithEvents`, or emits no event, is a bug тАФ wrap it. **A second, subtler bug:** a handler that authorizes a mutation against project P but then emits the event with a *different* `projectId` (a free typed value, a stale variable, `null`) тАФ the SSE subscription filter (`routes/events.ts`) narrows by that label, so a mislabeled event is durably stored and replayed under the wrong scope. The label passed to `emitEvent` MUST be the resource's authorized scope, derived from the same `requireResource`/`canSeeProject` decision that let the write through тАФ not re-supplied independently. *(Sharpened 2026-06-06 after the SSE-authorization audit: subscription-time filtering blocks the *leak* but not the *mislabel*; the convergence point owns atomicity AND label-fidelity. Residual is LOW тАФ filtering catches downstream тАФ but the check belongs at every `emitEvent` call site, not only at subscription.)*

6. **Data access (web): all HTTP goes through the one `client`, and react-query keys come from the per-resource key factories.** Converges on `apps/web/src/lib/api/client.ts:48` (`client`, ~25 importers) + 19 `*Keys` factory objects. A bare `fetch` outside the client, or an ad-hoc string query-key that should use a factory, is a bug тАФ drift risk on invalidation. *(Known soft spots: 3 sites hand-build literal keys that mirror a factory тАФ `documents.ts:190`, `projects.ts:78`, `routes/w.$wslug.tsx:301` тАФ tracked in `tasks/retro-follow-ups.md`.)*

7. **Token authority never exceeds the minting caller's role.** Converges on `roleToScopes` (`apps/server/src/lib/agent-schema.ts:118`) тАФ the roleтЖТscope ceiling. A write path that accepts caller-supplied scopes (or any authority) without intersecting `roleToScopes(role)` is a bug тАФ route it through the ceiling. *(This is the invariant whose bypass at `routes/tokens.ts` was the 2026-06-01 CRITICAL privilege-escalation fix `9f75c40`.)*

8. **Live updates: SSE teaches react-query WHEN to refetch; the event stream is not a source of truth EXCEPT for the two ratified live-tail/broadcast consumers below.** Converges on `apps/web/src/lib/api/event-stream.ts:49` (`useEventStream` тАФ docstring: "owns NO state and is NOT a source of truth"), 7 consumers. The default: events invalidate queries; the canonical GET refetches. A NEW consumer that builds UI state directly from an event payload (instead of invalidate-and-refetch) is a bug тАФ it desyncs тАФ UNLESS it is a broadcast-only datum with no GET, or an append-only live-tail feed, in which case it must be added to Deliberate exceptions (two such consumers are ratified there today). SSE filters match by **id**, cache keys by **slug** тАФ also see Deliberate exceptions.

9. **Error handling: failures throw `HTTPError` тЖТ the `{ error: { code, message } }` envelope тЖТ the client surfaces via `formatApiError`.** Converges on `apps/server/src/lib/http.ts:25` (envelope) + `apps/web/src/lib/api/errors.ts:23` (`formatApiError`, 27 web files). A handler that returns a bespoke error shape, or swallows an error (`catch {}` discarding it), or a UI that hand-formats an error instead of `formatApiError`, is a bug.

10. **Entity modeling: new entity types and fields are DATA (frontmatter) before they are tables.** Converges on the `documents` schema (`apps/server/src/db/schema.ts:232-237`): 3 typed columns (`title`, `status`, `body`) + a `frontmatter` JSON blob; "everything else lives in frontmatter for maximum flexibility." Agents, triggers, and (Phase 3) skills/memory are `type:'agent'`/`'page'` documents, NOT new tables. A migration adding a column for what is clearly an attribute of one type тАФ or a new table for what could be a document type тАФ is a bug. Re-confirm before adding any `*.sql` that adds a column.

11. **Skill trust: the `trusted` flag is a TYPED COLUMN on `instance_skills`, set ONLY via `setSkillTrust` (gated by `canBlessSkill`).** Converges on `apps/server/src/lib/skill-trust.ts` (`canBlessSkill` тАФ bless iff session user OR system-origin operator `createdBy IS NULL`; `setSkillTrust` тАФ the one sanctioned mutator, which flips the `instance_skills.trusted` column). `trusted` is load-bearing: a blessed skill loads as TRUSTED INSTRUCTIONS into agent system prompts (`runner.ts` `buildSkillsPreamble`), an unblessed one rides the untrusted DATA envelope. **The forging path is closed STRUCTURALLY (Phase 4, drop-workspace-tenancy): because `trusted` is its own column тАФ NOT a frontmatter key тАФ a body/frontmatter write (import, restore, clone, edit, a raw `.update(instanceSkills).set({frontmatter})`) physically CANNOT reach it.** This supersedes the prior `__system`-skills-page model where `trusted` rode frontmatter and had to be defensively stripped on every write (`stripManagedSkillTrust`/`isSystemSkillPage` тАФ both DELETED in Phase 4). The seeder (`instance-skills.ts`) strips any `trusted` key out of stored frontmatter and sets the column directly. The only way to bypass would be a NEW write surface that touches the `trusted` *column* outside `setSkillTrust` тАФ there is none today. Additionally, `set_skill_trust` carries `unattendedFloor: true` so trust-elevation is REFUSED on an unattended (trigger-fired) run at the `executeTool` convergence point (`agent-tools.ts` тАФ `UNATTENDED_FLOORED_SCOPES` OR `def.unattendedFloor`): the operator token is `createdBy`-null so `canBlessSkill` would otherwise pass even on a no-human run over attacker-supplied content. *(Ratified 2026-06-03 on the Piece-B diff; re-pointed to the typed `instance_skills.trusted` column 2026-06-05 in Phase 4 тАФ the structural guarantee REPLACES the strip-on-write defense; the `skill.trust.changed` event was dropped with the `__system` teardown.)*

12. **Irreversible-op confirmation is a property of the OPERATION (its risk tier), not of the caller's SURFACE.** An irreversible/high-risk tool call must be gated by an explicit human confirmation step BEFORE it executes, and which calls need that gate is decided in ONE place by the operation's risk tier тАФ never inferred from "which entry point is this." Converges on `apps/server/src/lib/agent-tools.ts` (`effectiveRiskTier(def)` тАФ write/delete scopes default to `high`; the `pending_ops` record + `choice_card` emit + confirm-and-replay-recorded-params flow at `executeTool`'s Task-7 gate). A tool whose risk tier should require confirmation but executes without one тАФ OR a new high-risk tool that defaults to `low` because nobody set its tier тАФ is a bug: tier the operation, don't trust the surface. **KNOWN GAP тАФ the gate is currently surface-coupled, which is the thing this invariant exists to drive OUT.** The confirm flow engages only when `caller.conversationId` is set (the operator cockpit chat); a **headless MCP `tools/call`** carries no `conversationId`, so the gate is SKIPPED and a high-risk op falls back to scope-only authority. Today's defense for the headless path is therefore invariant 2 (scope) + invariant 7 (role ceiling) ALONE тАФ a leaked MCP token with `agents:write`/`config:write` gets unattended high-risk writes with no confirmation. This is documented-acceptable for trusted admin automation TODAY, but it is a surface deciding a property the operation should own. The convergence target is: make `effectiveRiskTier` the sole decider on EVERY path (conversation, headless MCP, trigger-fired), with the *response* to a needed-but-impossible confirmation (no human present) being a fail-closed REFUSAL тАФ the way `unattendedFloor` already refuses trust-elevation on a no-human run (invariant 11) тАФ rather than a silent fall-through to scope-only. *(Named 2026-06-06 after the MCP-authorization audit. Listed under enforced invariants because the convergence point тАФ `effectiveRiskTier` at `executeTool` тАФ EXISTS and is correct; the gap is that one surface bypasses it. Track the close in `tasks/retro-follow-ups.md`; until then, the headless bypass is a Deliberate exception below.)* **The "needs approval" outcome is a CLEAN PAUSE, not a failure:** the gate emits the card + records the pending_op, then throws `AwaitingConfirmationError` (a typed signal, NOT `forbidden:`), which the runner catches BEFORE `isFatalToolError` and routes to a clean turn-end (`postResultAndComplete`, slot released) тАФ the same boundary `ask_choice` gets. The user's "Yes, do it" starts a fresh turn that re-runs the recorded params (M6). A true scope denial still throws `forbidden:` and stays fatal. *(Fixed 2026-06-06 after the real-BYOK "set up a CRM" smoke showed the pre-fix `forbidden: тАж requires confirmation` throw failing the run with `provider_error` тАФ which blocked the keystone "set up X for me" demo at the first config-write.)*

13. **Agent-doc resolution from a token: `token.agentId тЖТ agent Document` is decided in ONE place, which resolves the operator via its un-forgeable marker.** Converges on `apps/server/src/lib/agent-tools-registry.ts` (`resolveAgentDocForToken`) тАФ used by all 7 registry resolution sites (project-resolve, comment-author, describe_workspace, list_projects, find_documents, get_agent_self, the run-tools allow-list). The operator is a CODE SINGLETON with NO `documents` row. It is detected by the explicit **`token.isOperator` marker** on its `EphemeralToken` (`db/schema.ts`) тАФ NOT a DB column, so no persisted or forged token row can ever carry it; it is set ONLY in `createConversationRun`. This replaces the old FK-shaped `OPERATOR_AGENT_ID` sentinel-in-`agentId` (Shape BтА▓, 2026-06-07: that sentinel is gone тАФ the operator's token carries `agentId: null`). It is ALSO distinct from `isOperatorToken` (the unrelated `createdBy`-based helper) тАФ resolution keys on `token.isOperator` and NOTHING else. When `token.isOperator` is set the helper returns `getOperatorDocument()`; any other agentId still throws `agent_missing` if absent (real guard intact). A new site that open-codes `findFirst({id: token.agentId, type:'agent'})` is a bug тАФ route it through the helper, or it will mis-handle the operator. The operator is structurally UNREACHABLE on HTTP routes: the marker is not a column, so a `c.get('token')` materialized from `apiTokens.findFirst` can never bear it тАФ which is why the HTTP-surface `if (token.agentId)` branches need no operator handling at all. *(Named + completed 2026-06-06 in the cockpit shake-out тАФ the bug it fixes (operator `agent_missing` on every project-scoped tool) was caused by this resolution being scattered. ONE local exception, cycle-bound: `lib/agent-guards.ts` cannot import the registry helper тАФ registry imports agent-guards тАФ so it has its own `resolveCallingAgent`; both check `token.isOperator` FIRST тАФ keep the two in lockstep.) (sharpened 2026-06-07 тАФ Shape BтА▓: FK sentinel removed; operator identity = the non-persistable isOperator marker on EphemeralToken)*

14. **Skill rendering into the prompt: the trusted/untrusted skill body + label is rendered in ONE place, shared by every agent execution path.** Converges on `apps/server/src/lib/skill-preamble.ts` (`renderTrustedSkills`/`renderUntrustedSkills` + `TRUSTED_SKILLS_LABEL`/`UNTRUSTED_SKILLS_LABEL`) тАФ invariant 11's prompt-injection boundary depends on the wording matching across paths. The document path (`buildInitialMessages`), the conversation/cockpit path (`skillsToMessages`), and the cc path all render through these. A path that hand-rolls a skill label/body (bypassing the module) is a bug тАФ the wording would drift and a hardening of the untrusted-DATA framing wouldn't reach it. *(Named 2026-06-06 тАФ the absence of this convergence on the conversation path WAS the "operator doesn't follow its skill" bug; the cockpit path rendered no skill at all.)*

15. **FK-actor vs event-actor: a write's `created_by`/`updated_by` (FK тЖТ `users.id`) and its `events.actor` are resolved separately тАФ the FK column gets a real user, the event actor keeps the agent slug.** Converges on `apps/server/src/lib/agent-tools-registry.ts` (`serviceActor(ctx)` тАФ returns the FK-valid real user `ctx.confirmerId ?? token.createdBy`, fail-loud on a non-FK id) + the `eventActor` arg threaded through `create/update/deleteDocument`. For an agent/operator run `ctx.actor` is the slug `agent:<slug>` тАФ NOT a `users.id` (writing it to an FK column is errno 787), but it MUST stay the event actor so the agent-chain autonomy gate (`isAgentOriginated` тЖТ `agent:` actor, invariant 7-adjacent) keeps firing. A write that puts `ctx.actor` in an FK column (787), or that lets the event actor become the human user for an agent write (defeats `FOLIO_AGENT_CHAINS_ENABLED`), is a bug. *(Named 2026-06-06 тАФ the FK-787 on operator writes drove this split. Residual closed 2026-06-07 тАФ `eventActor` is now a REQUIRED param on `create`/`update`/`deleteDocument` (`documents.ts`), so omitting it is a compile error; a write service can no longer silently re-collapse the two by forgetting it.)*

16. **Board-view persistence: a board's working state (group-by/sort AND manual `board_position` order) persists to its ACTIVE view whenever an `activeView` is resolved тАФ NOT gated on `?view=` URL-pinning.** Converges on `apps/web/src/components/kanban/board-controls.tsx` (`onGroupByChange`, `onSortChange` тАФ the SOLE writer of the board-controls bus + view persistence; KanbanView is a pure reader) for group-by/sort, and `apps/web/src/components/views/kanban-view.tsx` (`onDragEnd`) for manual order. This is a single shared TRIGGER RULE with two distinct WRITE TARGETS (it is deliberately NOT one mutation, because the two pieces live on two entities): (a) group-by + sort persist to the **`views`** row via `useUpdateView`; (b) manual `board_position` persists to the **`documents`** row via the document PATCH (`onDragEnd` тЖТ `dropSlotPosition`/`computeReorderPosition` тЖТ `rankBetween`). The prior consent gate (`isActiveViewUrlPinned`, persist only when opened via `?view=<id>`) was meant for *ad-hoc unpinned* state, but the seeded DEFAULT view IS the user's real working view тАФ so changes on it were silently lost on reload until this rule. A NEW board-state writer that re-introduces a `?view=`-pinned gate (so default-board edits don't persist), or routes a `documents` `board_position` write through `useUpdateView` (wrong entity), is a bug тАФ persist whenever `activeView` is resolved, to the entity that owns the datum. Note: TableView keeps its own `?view=`-pinned autosave gate (`table-view.tsx`) тАФ that is a deliberate separate decision, NOT this rule; do not converge them without intent. *(Named 2026-06-07, hardening-pass Cluster 4 тАФ 4b un-gated default-board group/sort persistence; 4c un-parked manual drag-reorder; the architecture review flagged that the convergence note had to land HERE, not only in code comments, so a future writer + the invariant-auditor see it.)*

17. **Root-of-trust is session-only; everything else is admin-PAT-reachable.** The *root-of-trust class* тАФ minting ANY token (instance OR per-workspace), creating user/owner accounts, and changing instance ROLE (promote/demote owner/admin) тАФ is reachable ONLY via a logged-in session, NEVER via a bearer token. AI/BYOK-key WRITE is in the same session-only class (secret-class); reading "is AI configured?" may be headless. Every OTHER operation is reachable by an admin PAT, gated on the relevant scope (`agents:write`/`config:write` = owner/admin via `roleToScopes`, invariant 7). Converges on `requireSessionUser` (`apps/server/src/middleware/auth.ts`), mounted on the root-of-trust routes (`routes/instance-tokens.ts`, `routes/tokens.ts` POST, `routes/auth.ts` register, `routes/instance-users.ts` role-change, `routes/instance-ai-keys.ts` write), for the session-only set; and on the shared agent-lifecycle predicate `mayManageAgentLifecycle` (`apps/server/src/lib/agent-guards.ts`) тАФ the single admin-PAT-allowing gate that BOTH `assertNotHumanPatForAgentLifecycle` (HTTP) and `assertMcpAgentLifecycle` (MCP, `apps/server/src/lib/mcp-errors.ts`) delegate to тАФ plus the `executeTool` scope check (`apps/server/src/lib/agent-tools.ts`, invariant 2) for the admin-PAT set. A NEW admin write surface is EITHER in the root-of-trust list (mount it `requireSessionUser`) OR admin-PAT-reachable (route it through the scope gate) тАФ there is no third option; a credential-minting / account-creating / role-promoting path reachable by a bearer is a bug. Rationale: a leaked admin PAT must not be able to mint fresh credentials, self-promote, or write provider secrets тАФ blocking exactly these bounds the blast radius even when everything else is headless. *(Named 2026-06-09, headless-Folio-via-MCP Phase 1. The agent-lifecycle half (`mayManageAgentLifecycle`) is the convergence point this phase introduced тАФ it deliberately moved agent CRUD OUT of the "all human PATs rejected" stance into "admin PAT allowed", an accepted loosening; see the spec's threat model + the Deliberate exception on the headless confirm gate, which D1 is the first feature to lean on.)*

18. **Web current-table resolution: "which table am I on" and the table-route navigation branch are each decided in ONE web place.** A project can own several tables (`work-items`, `roadmap`, `bugs`, тАж); the *default* table's slug is `'work-items'`. Converges on `DEFAULT_TABLE_SLUG` + `useCurrentTslug` (`apps/web/src/lib/default-table.ts`) for the default-slug constant + the current-table resolver, and on `resolveTableNav` + `resolveViewNav` + `activeTableFromPath` (`apps/web/src/lib/rail-nav.ts`) for the default-vs-`/t/$tslug` navigation branch. The rules: (a) the default-table slug is `DEFAULT_TABLE_SLUG` тАФ the web twin of the server's `DEFAULT_TABLE_SLUG` (`apps/server/src/lib/seed-project-defaults.ts`); a component that hardcodes the string `'work-items'` instead of importing the constant is a bug. (b) "Which table is the current view on" is `useCurrentTslug`, which reads the route's `:tslug` param (the `/t/$tslug` routes) and falls back to `DEFAULT_TABLE_SLUG`; a component that re-derives the current table from the path itself, or hardcodes a slug where it has route context, bypasses this. Within one render subtree the table is resolved ONCE (via `useCurrentTslug` or a prop) and threaded тАФ a child re-reading the hook while a parent passes a prop is a split-brain bug. (c) The navigation branch (which route a table/view click targets) is `resolveTableNav`/`resolveViewNav`; the rail handlers (`apps/web/src/routes/w.$wslug.tsx`), the project-layout tabs (`apps/web/src/routes/w.$wslug.p.$pslug.tsx`), and the new-view sheet (`apps/web/src/components/views/new-view-sheet.tsx`) all delegate to them тАФ a fourth inline copy of the `isDefault ? /work-items : /t/$tslug` ternary is a bug. The three genuinely table-less surfaces (the root-mounted Cmd-K palette, the wiki page tree, the wiki-link picker) explicitly pass `DEFAULT_TABLE_SLUG` and are the documented exception, NOT a fourth resolution strategy. *(Named 2026-06-09, multi-table-UI feature. The bug class this prevents: the server is fully table-aware but the web silently routes/reads/writes against `work-items` regardless of the table the user is viewing тАФ "click bugs, see/write work-items". Adjacent to invariant 16: a multi-table board must persist group-by/sort/`board_position` to the ACTIVE table's view/docs, satisfied once the board's `useUpdateView`/`useUpdateDocument` are table-scoped тАФ verify a `bugs` board never writes to `work-items`' view.)*

## Open тАФ NOT yet converged (gaps)

- **Live-update connection model:** `useEventStream` opens ONE `EventSource` per consumer (7 today), not a shared workspace stream. This is a deliberate v1 simplicity choice, NOT a missing convergence point тАФ but it means "the live stream" is not yet a single connection. Tracked in `tasks/retro-follow-ups.md` (SSE fan-out, 2026-06-01) with an upgrade trigger (shared pool) if many panels coexist. Recorded here so it isn't mistaken for invariant 8 being violated.

- **MCP credential lifecycle:** MCP clients authenticate with a STATIC long-lived bearer token (`routes/mcp.ts` `requireToken`) тАФ there is no OAuth-style / capability-handshake / short-lived-grant flow. This is NOT a gap in *enforcement* тАФ the token's scopes route through the same `executeTool` double-check as every other path (invariant 2), so a stolen MCP token can do no more than its scopes allow. It is a gap in *credential lifecycle*. **The TTL half is now CLOSED (2026-06-07, hardening pass):** `api_tokens.expires_at` (nullable; null = never expires, existing tokens grandfathered) is enforced at `middleware/bearer.ts` `attachToken` тАФ an expired token collapses to the unknown-token path (token=null, same 401 shape, no expired-vs-unknown oracle), and `mintToken`/the create routes accept an optional `expires_in_days`. **Still deferred:** no rotation handshake (rotation today = revoke-old + mint-new, surfaced as a UI "Rotate" action тАФ no grace-window successor), no per-session capability narrowing, no revocation-on-use. SAFE today because tokens are scoped + role-ceilinged (invariants 2/7) and minting is gated. WATCH-ITEM for the remaining halves, not a fix-now: the MCP ecosystem is moving toward OAuth-style auth, and Folio's "agent is just another scoped API consumer" model is the right base to add it onto when MCP usage broadens beyond trusted first-party clients. Recorded here (2026-06-06, MCP-authorization audit; TTL closed 2026-06-07) so the lifecycle question isn't forgotten тАФ and so it's not mistaken for invariant 2 being unenforced (it is enforced).

*(All other audited properties are converged. No faked invariants.)*

## Deliberate exceptions (intentional bypasses тАФ do not re-flag)

- **`workspace_access` / `project_access` / `instance_skills` are real relational TABLES, not `documents` rows (invariant 10)** тАФ a justified exception to "entity types are DATA before tables." The two access tables are pure AUTHORIZATION data: a `(user, scope)` composite-PK join with a reverse index (`who can see this workspace?`), queried set-based on the hot path (every workspace/project-scoped request via `visibleProjectIds`/`canSeeWorkspace`). Modeling a grant as a `document` would make the convergence-point queries (joins, `IN`, existence checks) impossible. `instance_skills` is a typed table because its `trusted` flag MUST be a column, not frontmatter, to make trust-forging structurally impossible (invariant 11). These are correctly tables. (`apps/server/src/db/schema.ts` тАФ `workspaceAccess`, `projectAccess`, `instanceSkills`.) *(Ratified 2026-06-05 in Phase 4 тАФ recorded so the data-before-tables rule isn't re-flagged against them.)*
- **`useReactorHealth` writes the query cache directly (`setQueryData`) instead of invalidate-and-refetch (invariant 8)** тАФ because reactor health is broadcast-only; there is NO GET endpoint, so the event IS the only source of truth for that one datum. (`apps/web/src/lib/api/provider-health.ts`.)

## How to use this doc

- **`/code-review` + `/shakeout`:** verify the diff against these invariants. FLAG (don't block) any path that bypasses a convergence point as a finding: "bypasses invariant <N> (<property>) тАФ confirm intentional or route through <symbol>." - **Plan-writers (`superpowers:writing-plans` + `netdust-core:architecture-invariants`):** a plan touching one of these properties cites the invariant in a `## Architecture invariants touched` note, so the implementer routes through the convergence point. (Phase 3 `folio_api` already does this тАФ it rides invariants 2/3/4/7 via `app.request`, deliberately NOT hand-rolling auth.)
- **The built-in operator agent (Phase 3+):** reads this doc before editing Folio's own code/config. Staying inside the invariants is what makes the agent safe to let touch the app.
- **Keep it current:** add an invariant when a new convergence point is introduced; move a gap to Invariants when closed. A `/evaluate` retro that finds a bypass this doc didn't name should sharpen an invariant.
- **Traceability check (`bun run check:invariants`):** a cheap always-on layer that verifies each invariant's `Converges on` clause still POINTS AT REAL CODE тАФ the cited `file.ts` exists, the named symbols are still defined there, and any embedded `:NN` still matches. It does NOT verify the property holds (that is the `invariant-auditor` agent's job via `/shakeout`); it only catches the doc drifting off its anchors. Errors (missing file/symbol) exit non-zero; line drift warns. The pre-commit hook (`scripts/hooks/pre-commit-invariants.sh`, installed by `scripts/hooks/install.sh`) runs it non-blocking when this doc or server/web source is staged. When you move a cited symbol, refresh the `:NN`; when you rename/remove one, fix the citation here.
