## 1. Goal

Land the data + UX foundation for agent collaboration **without any AI key dependency**. After Phase 2.6 ships:

- A new `comment` document type exists. Humans (and agents authenticated via bearer tokens) can post comments on `work_item` and `page` documents.
- Every slideover renders a tabbed shell: **Fields В· Comments В· Activity** for work_item/page; **Fields В· Activity В· Runs** for agent/trigger.
- A Milkdown-lite comment composer with `@`-mention picker (agents allow-list-filtered + workspace members) ships in the Comments tab.
- The mention parser emits `comment.mentioned` events for resolved agent mentions, and detects approval keywords (`approved`/`rejected`) attached to a mention.
- Four built-in triggers auto-seed per workspace and visibly back the (currently inert) assignment + mention + approval + rejection flows. They become *active* in Phase 3 when the runner consumes them.
- The workspace agent slideover gains an Activity tab with the existing Phase 1.7 ActivityPanel + LogActivity wired in (Phase 2.5 deferral resolved).
- The trigger slideover gets a structured form (cron + event-kind dropdown + JSON payload editor) replacing the generic FrontmatterForm.
- Four new agent-lifecycle MCP tools (`create_agent`, `update_agent`, `delete_agent`, `get_agent_self`) ship under a new `agents:write` scope.
- An allow-list reconciler periodically scrubs orphan project ids from workspace agents' allow-lists.

What does NOT land in Phase 2.6:
- No AI provider abstraction. No agent runner. No `agent_run` documents. No `kind=plan/note/result/error` writing by a runner вҖ” only humans (and bearer-token agents) post in this phase.
- No `run_agent` MCP tool. No `[[` wiki-link picker integration with the body editor (only the comment composer ships it вҖ” body integration is Phase 3 polish).
- No templates surface вҖ” that's Phase 2.7.

The Phase 2.6 acceptance can be exercised end-to-end with no provider keys configured. The comments thread renders, agents can post via MCP using their bearer tokens, humans use the UI; approval buttons post the right comments but no runner exists yet to resume.

## 2. Data model

### 2a. `comment` value on `documents.type`

Migration `0007_phase_2_6_comments.sql` widens the existing enum via the SQLite table-rebuild idiom (same pattern as migrations 0005 + 0006). The check constraints become:

```
CHECK (
  (type = 'comment'                          AND parent_id IS NOT NULL AND table_id IS NULL) OR
  (type = 'agent'                            AND project_id IS NULL    AND parent_id IS NULL) OR
  (type = 'trigger'                          AND project_id IS NULL    AND parent_id IS NULL) OR
  (type IN ('work_item','page')              AND project_id IS NOT NULL)
)
```

The parent-type constraint (`parent.type IN ('work_item','page')`) is enforced by service-layer validation, not SQL вҖ” the existing schema does not encode parent-type referential integrity, and adding it now would require a join in the CHECK which SQLite doesn't support.

New index:

```sql
CREATE INDEX documents_comments_idx
  ON documents(parent_id, created_at DESC)
  WHERE type = 'comment';
```

This is the only index the Comments tab and mention-listing endpoints hit.

### 2b. Comment row shape

A comment row reuses the existing `documents` table вҖ” same columns, same indexes. The discriminators:

| Column | Behavior for comments |
|---|---|
| `type` | `'comment'` |
| `parent_id` | Required. FK вҶ’ `documents.id`. Target must be `work_item` or `page` (service-layer check). |
| `workspace_id` | Required. Inherited from parent at insert. |
| `project_id` | Required for work_item-parented comments (inherited from parent); inherited for page-parented (pages always have `project_id` in the current schema). |
| `table_id` | Always `null` on comments. |
| `title` | Server-generated short label `"Comment by <author> at <iso>"` for tooltip + slug derivation. Never displayed in the UI. |
| `slug` | Server-generated `c-<nanoid(8)>`. Short prefix prevents thread comments from polluting the document slug namespace. |
| `status` | Always `null` on comments. Comments don't have status. |
| `body` | The comment markdown. |
| `frontmatter` | Holds `author`, `kind`, `mentions`, `edited_at`, `target_agent`, `run_id`, `deleted_at` (see 2c). |

### 2c. Comment frontmatter Zod

`apps/server/src/lib/comment-schema.ts`:

```ts
export const CommentKindSchema = z.enum([
  'comment',    // default; freeform comment from human or agent
  'plan',       // agent's plan before action (Phase 3 runner writes; Phase 2.6 stub)
  'result',     // agent's final result (Phase 3 runner writes)
  'error',      // agent's error report (Phase 3 runner writes)
  'approval',   // human (or other agent) approval of a pending plan
  'rejection',  // human (or other agent) rejection of a pending plan
  'reply',      // callcenter pack: explicit reply categorization
]);

export const ResolvedMentionSchema = z.object({
  target: z.string().regex(/^(user|agent):.+$/),
  resolved: z.boolean(),
  resolvedId: z.string().optional(),       // documents.id (agent) or users.id (user). Set only when resolved=true.
  resolvedType: z.enum(['agent', 'user']).optional(),  // Set only when resolved=true.
});

export const CommentVisibilitySchema = z.enum(['normal', 'internal']);

export const CommentFrontmatterSchema = z.object({
  author: z.string().regex(/^(user|agent):.+$/),
  kind: CommentKindSchema.default('comment'),
  visibility: CommentVisibilitySchema.default('normal'),
  mentions: z.array(ResolvedMentionSchema).default([]),
  edited_at: z.string().datetime().optional(),
  target_agent: z.string().optional(),
  run_id: z.string().uuid().optional(),
  deleted_at: z.string().datetime().optional(),
})
.refine(
  d => !(d.kind === 'approval' || d.kind === 'rejection') || d.target_agent,
  { message: 'target_agent is required when kind is approval or rejection' }
)
.refine(
  d => !d.target_agent || d.kind === 'approval' || d.kind === 'rejection',
  { message: 'target_agent is only valid when kind is approval or rejection' }
);
```

The `mentions` array stores resolved targets at write-time. The `target` string is the human-readable form (`agent:<slug>` or `user:<id>`) preserved verbatim; `resolvedId` + `resolvedType` carry the database id (documents.id for agents, users.id for users) so the Phase 3 runner and the SSE filter pipeline can do direct id lookups without re-resolving the slug. If an agent is renamed or deleted later, the old mention's `target` slug + `resolvedId` are preserved; `resolved` is recomputed false on read when the id no longer maps to a live row. UI renders unresolved mentions as muted/strikethrough chips.

`deleted_at` is set on soft-delete; UI renders deleted rows as muted single-line ("Comment by @stefan В· deleted В· 5m ago"). No body content shown.

`visibility` discriminates operator-internal diagnostic comments (anonymizer logs, debug notes, agent-progress chatter that the reviewer doesn't need during triage) from normal user-facing comments. Default is `normal` вҖ” agents writing diagnostic output explicitly set `visibility: 'internal'`. The Comments-tab UI hides `internal` rows by default and offers a toggle to reveal them (В§6b). Server-side filtering happens via a new `?visibility=` query parameter on the list endpoint.

## 3. Services + routes

### 3a. `services/comments.ts`

Five exported functions:

```ts
createComment(tx, {
  workspaceId, projectId, parentId, authorContext,
  body, kind, targetAgent
}): Promise<Comment>

updateComment(tx, slug, { authorContext, body? }): Promise<Comment>
  // author-only; body change re-parses mentions, sets edited_at

deleteComment(tx, slug, { authorContext }): Promise<Comment>
  // author-only; soft delete (sets deleted_at, blanks body)

getComment(tx, slug): Promise<Comment | null>

listComments(tx, { parentId, kind?, since? }): Promise<Comment[]>
  // newest-first
```

Every mutation goes through the existing transactional `emitEvent(tx, args)` helper. `createComment` and `updateComment` parse mentions inside the same transaction and emit `comment.mentioned` per resolved agent mention.

`authorContext` is `{ type: 'user', userId } | { type: 'agent', agentSlug }`. Resolved by the route handler from the auth middleware before calling the service: session auth вҶ’ `{ type: 'user', userId: ctx.user.id }`; bearer auth with `agent_id` set on the token вҶ’ `{ type: 'agent', agentSlug: <resolved> }`; bearer auth without `agent_id` (human PAT) вҶ’ `{ type: 'user', userId: <token.user_id> }`. The route layer is the only place that knows about HTTP auth; the service receives the resolved actor.

### 3b. `lib/mention-parser.ts`

Pure function (testable in isolation). Implementation contract:

```ts
export function parseMentions(input: {
  body: string;
  workspaceAgents: { id: string; slug: string; allowedProjectIds: string[] | ['*'] }[];
  workspaceMembers: { id: string; email: string }[];
  currentProjectId: string;
}): {
  mentions: ResolvedMention[];   // includes resolvedId + resolvedType when resolved=true
  approvalIntent: { kind: 'approval' | 'rejection'; targetAgent: string; targetAgentId: string } | null;
}
```

#### Token regex

```ts
/(?:^|\s)@([a-z][a-z0-9-]+)/g
```

Matches `@<slug>` after whitespace or line start. The leading whitespace requirement prevents matching email addresses like `jan@example.com` mid-text.

#### Resolution

1. For each token, attempt to resolve as an agent:
   - Find agent with matching `slug` in `workspaceAgents`.
   - Check the agent's allow-list: `allowedProjectIds === ['*']` or `currentProjectId in allowedProjectIds`. If allow-list excludes the current project: `target: 'agent:<slug>', resolved: false` (agent exists but can't run here; no resolvedId stored since the runner shouldn't pick it up).
   - If allow-list permits: `target: 'agent:<slug>', resolved: true, resolvedId: <agent.id>, resolvedType: 'agent'`.
2. If not an agent, attempt to resolve as a member by email localpart:
   - Find members where `email.split('@')[0] === <slug>`.
   - If exactly one match: `target: 'user:<member.id>', resolved: true, resolvedId: <member.id>, resolvedType: 'user'`.
   - If zero or multiple matches: `target: 'user:<slug>', resolved: false`.
3. Deduplicate the result list by `target`.

The Phase 3 runner consumes `resolvedId` directly to find which run to resume (no slug-to-id query per event).

#### Approval keyword detection

After resolving mentions, scan the body for an approval intent attached to an agent mention. The rule (intentionally narrow):

- Find an agent mention (`target: 'agent:<slug>', resolved: true`).
- Look at the words immediately following the `@<slug>` token, up to the first 10 tokens (whitespace-delimited).
- If one of those 10 tokens is exactly `approved` or `rejected` (case-insensitive, with optional trailing punctuation `.`, `,`, `!`):
  - `approved` вҶ’ `{ kind: 'approval', targetAgent: <slug> }`
  - `rejected` вҶ’ `{ kind: 'rejection', targetAgent: <slug> }`
- If no approval keyword found near any mention, return `approvalIntent: null`.
- If multiple mentions have approval intent, take the *first* one (deterministic).

Examples:
- `"@drafter approved вҖ” looks great"` вҶ’ approval intent for `drafter`.
- `"@drafter looks approved to me"` вҶ’ NO intent (`approved` is past position 1, not first-word-after-mention within the strict window? Actually wait: "looks" is word 1, "approved" is word 2 вҖ” both within 10 tokens. So this DOES match. We need a tighter rule.)

**Refined rule:** `approved` or `rejected` must be one of the **first two tokens** after the mention (i.e. immediately after the mention's whitespace boundary, or the second token). Anything beyond position 2 doesn't count.

Re-examples with refined rule:
- `"@drafter approved"` вҶ’ match (position 1).
- `"@drafter approved вҖ” looks great"` вҶ’ match (position 1).
- `"@drafter is approved"` вҶ’ match (position 2).
- `"@drafter looks approved to me"` вҶ’ no match (position 3+).
- `"@drafter please approve"` вҶ’ no match (verb form, not adjective).

This is documented in `docs/AGENTS.md` so reviewers know the rule.

### 3c. `routes/comments.ts`

Mounted under both project-scoped paths:

```
POST   /api/v1/w/:wslug/p/:pslug/documents/:parentSlug/comments
GET    /api/v1/w/:wslug/p/:pslug/documents/:parentSlug/comments?kind=...&since=...
GET    /api/v1/w/:wslug/p/:pslug/comments/:slug
PATCH  /api/v1/w/:wslug/p/:pslug/comments/:slug
DELETE /api/v1/w/:wslug/p/:pslug/comments/:slug
```

| Method | Behavior |
|---|---|
| `POST` | Body: `{ body: string, kind?: CommentKind, target_agent?: string }`. Server resolves `author` from auth. Server runs `parseMentions(...)`. If `parseMentions` returns an `approvalIntent`, that *overrides* the client's `kind` (client cannot post `kind=approval` without the keyword; keyword wins to prevent client/server mismatch). `target_agent` is set from `approvalIntent.targetAgent` if intent present. Server validates: parent type, body non-empty, body вүӨ 64 KB, `target_agent` rules. Inserts row, emits `comment.created` + per-mention `comment.mentioned`. |
| `GET (list)` | Returns parent's comments newest-first. `?kind=plan,result` (comma-separated set), `?since=<iso>`, and `?visibility=normal,internal` (comma-separated set; default is `?visibility=normal` so internal rows are excluded unless explicitly requested) filters. Comments with `deleted_at` set still return вҖ” UI handles the muted render. |
| `GET (single)` | Returns one comment. |
| `PATCH` | Only the comment's `author` can edit (user-id or agent-slug must match). `body` updates вҶ’ re-parse mentions, set `edited_at`. `kind` is **immutable** вҖ” 422 `KIND_IMMUTABLE` if attempted. Removed mentions don't fire anything; added mentions fire fresh `comment.mentioned` events. Approval intent on edit *adds* a new approval вҖ” but only if the comment was created with `kind=approval` already (you can't promote a plain note to an approval after the fact). Behavior: if the original comment was `kind=approval`, editing recomputes `target_agent` from the new body's intent; if the original was `kind=comment`, the keyword detection is logged but doesn't change `kind`. |
| `DELETE` | Only the author can delete. Sets `deleted_at`, blanks body to empty string. Emits `comment.deleted`. |

**Auth:**
- Session: user must be a workspace member.
- Bearer token: scope `documents:write` for mutations, `documents:read` for GETs. `requireResource()` enforces project allow-list for agent-bound bearer tokens (Phase 2.5 middleware).

**Validation errors:**
- `INVALID_COMMENT_PARENT` (422): parent doesn't resolve, is not `work_item`/`page`, or is in a different workspace.
- `EMPTY_COMMENT_BODY` (422): body trimmed to empty.
- `COMMENT_BODY_TOO_LARGE` (422): body > 64 KB.
- `TARGET_AGENT_REQUIRED` (422): kind=approval/rejection without target_agent (and no approval keyword in body to fill it in).
- `TARGET_AGENT_FORBIDDEN` (422): target_agent set on kind вҲү {approval, rejection}.
- `KIND_IMMUTABLE` (422): PATCH attempting to change kind.
- `COMMENT_AUTHOR_ONLY` (403): non-author trying to edit or delete.

## 4. Events + SSE

### 4a. New event kinds

Added to `apps/server/src/lib/trigger-schema.ts::KNOWN_EVENT_KINDS`:

| Kind | Payload | Fired when |
|---|---|---|
| `comment.created` | `{ document_id, parent_id, author, kind, target_agent? }` | New comment row inserts. Fires once per insert. |
| `comment.mentioned` | `{ comment_id, parent_id, agent_slug }` | A comment with one or more resolved agent mentions is inserted. Fires once per resolved agent (so a comment mentioning 3 agents emits 3 events). |
| `comment.deleted` | `{ document_id, parent_id, author }` | A comment is soft-deleted. |
| `agent.allow_list.reconciled` | `{ agent_id, removed_project_ids: string[] }` | Allow-list reconciler scrubs orphan ids from an agent. Fires per scrubbed agent (no event when reconciler runs but finds nothing to scrub). |

### 4b. SSE filter additions

`/api/v1/w/:wslug/events` gains two new query parameters:

- `?parent=<doc_id>` вҖ” events whose payload's `parent_id` matches. AND-combined with existing `?kinds=` + `?project=`.
- `?run=<run_id>` вҖ” events whose payload's `run_id` matches. Used in Phase 2.6 only by the approval-button consumer; agent runs themselves don't exist yet. The query parameter is still parsed and applied to comments, future-proofing for Phase 3.

The `Last-Event-Id` replay mechanism from Phase 2 works with all new event kinds.

## 5. MCP tools

Eight new tools shipped in Phase 2.6 вҖ” four for comments, four for agent lifecycle. The MCP server stays hand-rolled (no SDK dependency); each tool is added to the existing dispatch table in `apps/server/src/routes/mcp.ts`.

### 5a. Comment tools

| Tool | Scope | Args | Returns |
|---|---|---|---|
| `create_comment` | `documents:write` | `{ parent_slug, body, kind?, target_agent? }` | `{ slug, kind, target_agent? }` |
| `list_comments` | `documents:read` | `{ parent_slug, kind?, since? }` | `Comment[]` |
| `update_comment` | `documents:write` | `{ slug, body }` | `{ slug, edited_at }` |
| `delete_comment` | `documents:delete` | `{ slug }` | `{ slug, deleted_at }` |

`create_comment` for an agent token:
- `author` is resolved server-side to `agent:<slug>` via the bearer's `agent_id` column.
- Parent must resolve to a doc in the agent's allow-listed projects (Phase 2.5 `requireResource` semantics).
- Mention parsing applies (so an agent posting `@<other-agent> approved` legitimately approves another agent's pending run).

`update_comment` and `delete_comment`:
- Allowed only if the comment's `author` matches the calling token's `agent_id`.
- Other-author edit attempts return JSON-RPC error `-32603` with `data: { reason: 'comment_author_only' }`.

### 5b. Agent-lifecycle tools (Phase 2.5 deferral)

A new MCP scope `agents:write` is added to the vocabulary. Workspace settings UI gets a checkbox for it; existing Read+write and Full access presets include it.

| Tool | Scope | Args | Returns |
|---|---|---|---|
| `create_agent` | `agents:write` | `{ workspace_slug, slug, title, body, frontmatter }` | `{ slug, agent_token }` вҖ” token returned ONCE |
| `update_agent` | `agents:write` | `{ slug, title?, body?, frontmatter? }` | `{ slug }` |
| `delete_agent` | `agents:write` | `{ slug }` | `{ slug, deleted: true }` |
| `get_agent_self` | none | `{}` | the calling agent's full document |

`create_agent`:
- Same path as the HTTP `POST /api/v1/w/:wslug/documents` agent-create endpoint (Phase 2.5).
- Auto-mints an API token, returns it once in the response.
- The calling agent must have `agents:write` in its allow-listed tools. Default agents don't; meta-agents do.

`update_agent`:
- Allows changing frontmatter, body, title.
- **Allow-list narrowing rule:** if the update would change `frontmatter.projects`, the new value must be a subset (or equal) of the calling agent's *own* `projects` allow-list. This prevents a meta-agent in a narrow allow-list from creating an agent with broader access. Returns `-32602` with `data: { reason: 'allow_list_widening_forbidden' }` on violation.

`delete_agent`:
- Transactional revoke of the auto-minted token (existing Phase 2.5 cascade via `api_tokens.agent_id` FK).
- The cascade already drops the workspace's PATCH-based cleanup; nothing new here.
- Returns `-32602 cannot_delete_self` if the agent attempts to delete itself.

`get_agent_self`:
- No scope check (a calling agent can always inspect itself).
- Resolves via the bearer's `agent_id` column.
- Returns the agent document including frontmatter вҖ” useful for agents that branch behavior on their own configuration.

### 6b. Comments tab

Schema changes:

- `apps/server/src/lib/trigger-schema.ts` accepts `$event.<key>` syntax in the `agent:` field. Trigger-fire path (Phase 3 runner) resolves the reference at fire time.
- New optional field `internal_action: 'resume_run' | 'reject_run' | null`. When set, the trigger's action is an internal call (not `runAgent`). In Phase 2.6 the matcher recognizes these but the handlers are stubs (`resume_run` / `reject_run` log "Phase 3 wires runner"). Phase 3 implements the handlers.
- `frontmatter.builtin: true` boolean flag.
- New event kind `agent.task.assigned` already exists from Phase 2 вҖ” no change.
- New event kinds `comment.created`, `comment.mentioned` added in this phase (4a).

**Enable-default policy:**
- `builtin-on-assignment` and `builtin-on-mention` ship with `enabled: false` in Phase 2.6. They're visible in the trigger list with a muted "(activates in Phase 3 вҖ” agent runner)" banner. The operator can manually flip them on in 2.6 if they want the events to fire (no consumer, so this is informational only). Phase 3's migration auto-flips them to `enabled: true` workspace-by-workspace.
- `builtin-on-approval` and `builtin-on-rejection` ship with `enabled: true`. Their UI surface (the approval/rejection comment a human posts) exists in 2.6. Their `internal_action` handler is the stub that becomes real in Phase 3 (resume_run / reject_run).
- This avoids the "firing into the void" problem вҖ” Phase 2.6 doesn't burn cycles matching events against unhandled actions.

Server enforcement:
- PATCH on `builtin: true` trigger document: only the `enabled` field is mutable. Any other change вҶ’ 422 `BUILTIN_TRIGGER_LOCKED` with a list of fields that can be changed.
- DELETE on `builtin: true` trigger document вҶ’ 422 `BUILTIN_TRIGGER_LOCKED`.

Auto-seed lives in `services/workspaces.ts::createWorkspace` вҖ” after the existing workspace + initial-membership rows insert, four trigger documents are inserted in the same transaction.

Existing workspaces (pre-2.6 deploys) need backfill. A backfill script ships with the migration: `scripts/backfill-builtin-triggers.ts` вҖ” idempotent, inserts the four documents for every workspace that doesn't already have them. Stefan runs this once during the 2.6 deploy.

### 6g. Allow-list reconciler

`lib/reconciler.ts` вҖ” a periodic in-process job:

```ts
async function reconcileAllowLists(db: Database): Promise<void> {
  // For every agent in every workspace:
  //   Load frontmatter.projects (or default ['*'])
  //   If ['*'], skip (wildcard agents don't need reconciliation)
  //   Otherwise:
  //     For each project_id in projects:
  //       If not exists in projects table вҶ’ mark as orphan
  //     If any orphans: PATCH agent.frontmatter.projects = filter(p => p not in orphans)
  //                      emit agent.allow_list.reconciled { agent_id, removed_project_ids }
}
```

Scheduled via a simple `setInterval` at server boot:

```ts
// apps/server/src/index.ts
const intervalMs = Number(process.env.FOLIO_RECONCILER_INTERVAL_MS ?? 60 * 60 * 1000); // 1h default
setInterval(() => reconcileAllowLists(db).catch(logError), intervalMs);
```

Single transaction per agent (atomic update + event emission).

The Phase 2.5 project-delete cascade already scrubs orphans transactionally (per `memory/DECISIONS.md`). The reconciler is the insurance: catches hand-edited MD imports, partial restores from backup, and any future bug in the cascade that lets orphans slip through.

Logs: at INFO level, every run logs `reconciled: <count>` agents touched. No log when zero agents needed scrubbing.
