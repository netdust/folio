# Phase 2.5 — Workspace-Scoped Agents (Design)

**Date:** 2026-05-26
**Status:** Approved — pending implementation plan
**Supersedes:** Phase 2 project-scoped agent surface
**Companion locked decisions:** `memory/DECISIONS.md` → "Phase 2.5 — Agent scope model (2026-05-26)"

## Purpose

Phase 2 shipped agents and triggers as project-scoped documents. Operational reality (one user, three workspaces, sharing the same "Triage Bot" across projects in a workspace) makes per-project duplication of agents painful. This phase moves agent + trigger identity to the workspace level with an explicit `projects:` allow-list in frontmatter, removes the project rail leaves, and surfaces both from the existing workspace popover.

The model is grounded in a research pass across GitHub Apps, Slack, Linear, Notion, MCP spec, Cloudflare/AWS/Vercel tokens, macaroons/biscuits, and Zanzibar-style ReBAC. Convergent findings:

- Separate principal from credential (GitHub Apps three-layer model).
- Default-deny on cross-project access with explicit grants (Notion).
- Action-scope and resource-scope are orthogonal in middleware (AWS/Cloudflare/GCP).
- Live re-eval per request against SQLite, no JWT (revocation is the #1 operator ask).
- Skip macaroons (Fly.io: end-users don't attenuate in practice on a single-binary deploy).
- Skip ReBAC (violates no-sidecar rule; permission shape here is a flat allow-list).

## Non-goals

- **Cross-workspace agent identity.** Workspace-as-tenant invariant stays clean. Templates (Phase 2.6) cover the "edit once" workflow without breaking it.
- **Project-scoped agent variant alongside workspace-scoped.** Two mental models would be tax for zero capability gain. Workspace-scoped with `projects: ['<one>']` is identical in behavior to project-scoped.
- **Macaroons / Biscuit / ReBAC / external authz services.**
- **Per-project action-scope overrides** (read on A, write on B). If the use case emerges, create two agents.
- **Templates** — separate Phase 2.6 spec.
- **Agent-lifecycle MCP tools** (`create_agent`, `update_agent`, `delete_agent`, `get_agent_self`, single-project `project_slug` inference) — Phase 2.6.

## Decisions locked

1. **Agents live at workspace level.** `documents.workspace_id NOT NULL`. `documents.project_id NULL` allowed iff `type IN ('agent','trigger')`. Enforced via CHECK constraint.
2. **Project binding is frontmatter.** New field `projects: string[]` on agent + trigger frontmatter. Default `['*']`. Values are project **ids** (uuids), not slugs — survives project renames without rename-time fixup.
3. **`'*'` is exclusive.** Mixing wildcard with explicit ids (`['*', 'abc']`) is rejected at Zod parse time.
4. **Triggers inherit agent's allow-list.** Trigger has no `projects:` field. Fires once per project in the referenced agent's allow-list.
5. **Tokens carry `agent_id` and optional `project_ids` (narrowing only).** Effective allow-list = `intersect(agent.frontmatter.projects, token.project_ids ?? null)`. Tokens may narrow agent's bounds; never broaden.
6. **Live re-eval per request.** Opaque hashed tokens, no JWT. Revocation = `revoked_at` flag, next request dies.
7. **Action-scope and resource-scope orthogonal.** Existing `requireScope(verb)` keeps the action check. New `requireResource(req)` is the resource check. Both must pass; never merged into a single scope string.
8. **Rail subtraction.** Agents and Triggers rail leaves removed from every project. Rail rule becomes: rail = content (workspaces, projects, tables, views, wiki); header = infrastructure (agents, triggers, settings).
9. **Workspace popover gains Agents + Triggers entries.** Above the existing footer (`+ New project`, `+ Create workspace`, `Workspace settings`).
10. **MCP list_projects filters by allow-list.** Agents only see projects they can act on. Aligns with Notion's default-deny philosophy.
11. **MCP create/update/delete for agent docs rejected in this phase.** Agent-lifecycle MCP tools ship in Phase 2.6.
12. **No data migration.** No agents are deployed. Structural schema migration only; demo seed drops any existing agent rows and re-seeds at workspace level if seeding is run.
13. **Project-delete cascades through agent allow-lists.** Application-level hook on `DELETE /projects/:id` scans workspace agents and removes the deleted project's id from their `frontmatter.projects` arrays. SQLite can't enforce FK across JSON columns; this is the enforcement point.

## Data model

### `documents` table

```diff
- project_id text NOT NULL REFERENCES projects(id) ON DELETE CASCADE
+ project_id text REFERENCES projects(id) ON DELETE CASCADE          -- nullable
+ workspace_id text NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE
+ CHECK (
+   (type IN ('agent','trigger') AND project_id IS NULL)
+   OR
+   (type IN ('work_item','page') AND project_id IS NOT NULL)
+ )
```

Backfill flow (SQLite can't `ALTER COLUMN` to set NOT NULL; uses the CREATE+COPY+DROP+RENAME idiom established in `0003_phase_2a_tables.sql`):

1. Add `workspace_id` nullable to `documents`.
2. `UPDATE documents SET workspace_id = (SELECT workspace_id FROM projects WHERE id = documents.project_id)`.
3. Defensive: assert `SELECT COUNT(*) FROM documents WHERE workspace_id IS NULL = 0`. If any rows remain (orphan from a non-cascading delete somehow), fail the migration loud — don't silently coerce. Operator decides whether to drop the orphans or recover their project.
4. Rebuild `documents` table via `CREATE TABLE documents_new (...)` with `workspace_id NOT NULL`, `project_id` nullable, the new CHECK, indexes; `INSERT INTO documents_new SELECT * FROM documents`; `DROP documents`; `ALTER TABLE documents_new RENAME TO documents`.

Indexes:
- Keep existing `(project_id, slug)` for work_item/page uniqueness.
- New `(workspace_id, type, slug)` UNIQUE for agent/trigger uniqueness.
- New `(workspace_id, type)` for the workspace-level listing endpoints.

### `api_tokens` table

```diff
+ agent_id text REFERENCES documents(id) ON DELETE CASCADE   -- nullable
+ project_ids text                                            -- nullable JSON array
```

- `agent_id` null = human PAT.
- `project_ids` null = inherit from agent (or, for human PATs, unbounded within workspace).
- `project_ids = []` = effectively revoked at the resource layer.
- When set, `project_ids` must be subset of `agent.frontmatter.projects` (validated at token-create time, 422 if broader).

### Frontmatter schemas

Agent (`packages/shared/src/agent-frontmatter.ts`, or existing equivalent) gains:

```ts
projects: z.array(z.string()).default(['*'])
  .refine(arr => !(arr.includes('*') && arr.length > 1),
          { message: "'*' cannot be combined with explicit project ids" })
```

Trigger schema is unchanged — triggers do not carry a `projects:` field.

## Middleware composition

```
Request → resolveWorkspace
       → resolveProject (only if URL has :pslug)
       → auth (session OR bearer)
       → requireScope(action)
       → requireResource(req)        ← NEW
       → handler
```

`requireResource(req)` runs only for bearer requests. Session users bypass the *deny* check — their workspace membership is the gate for access. The server-side *filter* (`?project=<id>` narrowing the agent list to those allow-listed for that project) still applies to session-driven UI queries; that filter is a list-shape decision, not an authorization decision.

```ts
async function requireResource(c, next) {
  const token = c.get('token');
  const proj  = c.get('project');   // null if no :pslug

  if (!proj) return next();         // no resource to check

  if (token.agent_id) {
    const agent = await loadAgent(token.agent_id);
    const effective = intersect(agent.frontmatter.projects, token.project_ids ?? null);
    if (!effective.includes('*') && !effective.includes(proj.id)) {
      throw new HTTPError('FORBIDDEN_RESOURCE',
        `agent not allow-listed for project ${proj.slug}`, 403);
    }
  }
  return next();
}
```

`intersect(agentList, tokenList)`:
- `(['*'], null)` → `['*']`
- `(['*'], ['a','b'])` → `['a','b']`
- `(['a','b','c'], ['b','c','d'])` → `['b','c']`
- `(['a','b'], null)` → `['a','b']`
- `(['a'], [])` → `[]`

**Performance:** one indexed PK lookup per agent-bearer request. No caching in v1; measure first.

## Routes

### New (workspace-level documents)

```
GET    /api/v1/w/:wslug/documents?type=agent             list workspace agents
GET    /api/v1/w/:wslug/documents?type=agent&project=:pid filter to allow-list
GET    /api/v1/w/:wslug/documents?type=trigger           list workspace triggers
POST   /api/v1/w/:wslug/documents                        create (type=agent|trigger only)
GET    /api/v1/w/:wslug/documents/:slug                  read
PATCH  /api/v1/w/:wslug/documents/:slug                  update
DELETE /api/v1/w/:wslug/documents/:slug                  delete (revokes auto-minted token)
```

### Changed (project-level documents)

- `POST /api/v1/w/:wslug/p/:pslug/documents` — rejects `type IN ('agent','trigger')` with `422 INVALID_DOCUMENT_SCOPE`. Message points to the workspace-level endpoint.
- `GET /api/v1/w/:wslug/p/:pslug/documents?type=agent` — rejects with `400 UNSUPPORTED_TYPE_FILTER`. Clients hit the workspace endpoint with `?project=:pid`.

### Token-create endpoint

`POST /api/v1/w/:wslug/tokens` body gains optional `project_ids: string[]`. When the caller is creating a token for an agent (server-side flow), `project_ids` must be subset of `agent.frontmatter.projects`; 422 otherwise.

### Unchanged

`/api/v1/w/:wslug/p/:pslug/{statuses,fields,views,tables}` — all stay project-scoped. Agents touch these through `requireScope` + `requireResource` like everything else.

## MCP changes

Server location and JSON-RPC shape unchanged. Resolver behavior:

`resolveProjectInWorkspace(ws, args)` — after slug → row resolution, check `token.agent_id`:
- Load agent, compute `effective = intersect(agent.projects, token.project_ids ?? null)`.
- If `!effective.includes('*') && !effective.includes(project.id)`: return MCP error `-32602` with `data: { reason: 'agent_not_in_allow_list', project_slug, agent_slug }`.

Per-tool:

| Tool | Phase 2.5 behavior |
|---|---|
| `list_workspaces` | Unchanged (single workspace bound to token). |
| `list_projects` | **Filtered by allow-list.** Wildcard returns all workspace projects; explicit list returns only those. |
| `list_documents` | Resolver enforces project membership. |
| `get_document` / `get_document_markdown` | Resolver enforces. |
| `create_document` | For `type='work_item'\|'page'` — resolver enforces. For `type='agent'\|'trigger'` — **rejected** (`agents must be created via workspace-scoped HTTP endpoint`). |
| `update_document` | For project docs — resolver enforces. For agent docs — **rejected** in this phase; Phase 2.6 enables. |
| `delete_document` | Same gating as `update_document`. |
| `list_statuses` / `list_fields` / `list_views` / `run_view` | Resolver enforces project. |

Error code chosen: `-32602` (invalid params). The agent is being asked to act on a project it has no parameter authorization for; clients handle `-32602` more gracefully than server-error codes.

### Workspace agents page — new route `/w/:wslug/agents`

Reuses `DocumentTypeList`. Per-row renderer adds project chips next to the agent name. Chip resolution: lookup `id → current slug` via the already-loaded `useProjects(wslug)` query.
- Wildcard `'*'` → single "All projects" chip.
- Specific ids → one chip per project, current slug rendered.
- Orphan id (project deleted, cascade hadn't run yet) → muted `<id-prefix>·removed` chip.

Interactions:
- Click a chip → filter list to agents touching that project (`?project=<id>` query).
- Click row elsewhere → opens slideover (existing Phase 2 behavior).
- `+ New agent` button: top-right of the page header (primary button styling). Also surfaced as the CTA on the empty state when no agents exist. Both open the slideover in create mode.

### Workspace triggers page — new route `/w/:wslug/triggers`

Same shape. Row shows `[name] [schedule pill] [agent name] [project chips]`. Chips are derived from the referenced agent's `frontmatter.projects` at render time.

### Agent edit slideover — `projects` field

New `ProjectsField` renderer in `FrontmatterForm`, auto-wired when `key === 'projects'`. Always multi-select:
- Multi-select Popover lists all workspace projects with a checkbox per project plus a "Select all" checkbox at the top. The "Select all" state binds to storing `['*']` in frontmatter; deselecting it stores the explicit id list.
- Selected projects render as removable chips inline (current-slug rendering).

### Assignee picker

`apps/web/src/components/assignee/assignee-picker.tsx` swaps from project-scoped `useDocuments(wslug, pslug, { type: 'agent' })` to workspace-scoped `useWorkspaceAgents(wslug, { project: projectId })`. ## Deferrals — Phase 2.6 (must not be forgotten)

> **BLOCKER for v1 ship:** The following are not optional features; they're sequenced after Phase 2.5 only because they pair naturally with templates. The product wedge is **agent-first lifecycle**, so agents being able to create/update agents through MCP is non-negotiable for v1.

- `create_agent` MCP tool — agents create new agents (often from templates). **Required for v1.**
- `update_agent` MCP tool — agents edit themselves and other agents (mostly for sync-from-template flows).
- `delete_agent` MCP tool.
- `get_agent_self` MCP tool — agent introspection (`{ slug, projects (as slugs), tools }`). Lets prompts say "If unsure which project to act on, call get_agent_self first."
- Single-project `project_slug` arg inference — agents whose `projects:` has exactly one id can omit `project_slug` and the server defaults to that project. Polish.
- Templates as a whole — instance-level Settings page, inert markdown, `template:` + `template_version:` references on instances, sync UI.

## Deferrals — Phase 3+ (non-blocking)

- Human PAT `project_ids` enforcement — schema column exists from Phase 2.5; enforcement waits until human PATs get a UI for narrowing.
- Per-project action-scope overrides — only if a real use case shows up.
- Caching the agent's `projects:` allow-list — measure `requireResource` perf first.
