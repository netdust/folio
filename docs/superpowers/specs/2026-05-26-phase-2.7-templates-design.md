## 1. Goal

Ship an instance-level templates catalog so reusable agents and triggers can be installed into workspaces with pinned-version sync. Workspaces stay bound to a specific template version; publishing a new version surfaces an "Update available" banner; updates are diff-then-confirm. Templates are full-stack-exposed: everything humans can do via UI, agents can do via MCP — events emit on every mutation. Templates aren't an admin-only walled garden; they're an instance-level surface gated by a new `templates:admin` scope.

After Phase 2.7 ships:

- **Templates and template_versions** live in two new database tables (instance-level, not workspace-scoped).
- **Agents AND triggers** are templatable — both document types covered.
- **`/settings/templates`** UI lets an admin browse the catalog, edit templates, publish new versions, install into workspaces.
- **Workspace agent + trigger documents** can carry optional `template: { id, version }` frontmatter; when present, the slideover surfaces an "Update available" banner if a newer version exists.
- **MCP exposes the full surface** — list, get, create, update, publish, delete templates, plus install / sync-to-workspace. Same scope-gated.
- **New scope `templates:admin`** — instance-level (not workspace-level), required for any template mutation via bearer auth.
- **Every mutation emits events** — `template.created`, `template.published`, `template.updated`, `template.deleted`, `template.installed_to_workspace`, `template.workspace_doc_updated`, `template.workspace_doc_orphaned`.
- **Diff machinery** preserves workspace-local edits on update when non-conflicting; surfaces conflicts for manual resolution.

What does NOT land in Phase 2.7:

- No bundled "pack" concept (e.g. "install the callcenter pack" as one click). Each template installs individually. Packs are Phase 7+ polish if customer pull surfaces.
- No template registry / cross-instance sharing. Source-of-truth is each instance's DB. Stefan hand-authors per install (or builds a future export/import tool).
- No automatic syncing on publish — every update is opt-in per workspace. "Auto-update workspaces on publish" is a future flag.

## 2. The principle

**Everything humans do via UI, agents do via MCP.** This applies to templates as much as to documents.

Concrete consequences:
- Every UI action has an HTTP endpoint behind it.
- Every HTTP endpoint has an MCP tool that calls the same service layer.
- Every mutation emits an event through the existing event bus.
- Scope gating is the only access boundary — humans use session auth, agents use bearer auth with the right scope.
- The `/settings/templates` admin page is gated by session auth OR `templates:admin` bearer.

Templates aren't a walled garden because there's no operational reason for them to be one. A meta-agent that bootstraps customer instances should be able to publish templates as easily as a human admin can.

## 3. Data model

### 3a. New tables

Two new tables. Both are instance-level — no `workspace_id` column. Templates exist outside the workspace hierarchy.

```sql
CREATE TABLE template_groups (
  id          TEXT PRIMARY KEY,                              -- uuid v7
  slug        TEXT NOT NULL UNIQUE,                           -- e.g. 'callcenter'
  title       TEXT NOT NULL,
  description TEXT,
  created_by  TEXT REFERENCES users(id) ON DELETE SET NULL,
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL,
  deleted_at  INTEGER
);

CREATE INDEX template_groups_slug_idx ON template_groups(slug) WHERE deleted_at IS NULL;

CREATE TABLE templates (
  id                TEXT PRIMARY KEY,                          -- uuid v7
  doc_type          TEXT NOT NULL CHECK (doc_type IN ('agent', 'trigger')),
  slug              TEXT NOT NULL UNIQUE,                       -- human-friendly identifier
  title             TEXT NOT NULL,
  description       TEXT,
  template_group_id TEXT REFERENCES template_groups(id) ON DELETE SET NULL,  -- nullable; forward-compat for Pack installs (Phase 7+)
  current_version   INTEGER NOT NULL DEFAULT 0,                 -- 0 = no version published yet (draft only)
  created_by        TEXT REFERENCES users(id) ON DELETE SET NULL,
  created_at        INTEGER NOT NULL,
  updated_at        INTEGER NOT NULL,
  deleted_at        INTEGER                                     -- NULL = live; non-null = orphaned-workspace state (see 3c)
);

CREATE INDEX templates_doc_type_idx ON templates(doc_type) WHERE deleted_at IS NULL;
CREATE INDEX templates_slug_idx     ON templates(slug)     WHERE deleted_at IS NULL;
CREATE INDEX templates_group_idx    ON templates(template_group_id) WHERE deleted_at IS NULL;

CREATE TABLE template_versions (
  id                  TEXT PRIMARY KEY,                              -- uuid v7
  template_id         TEXT NOT NULL REFERENCES templates(id) ON DELETE CASCADE,
  version             INTEGER NOT NULL,                              -- monotonic per template; v1, v2, v3...
  title               TEXT NOT NULL,                                 -- snapshot of template.title at publish time
  body                TEXT NOT NULL,                                 -- markdown body (system prompt for agents)
  frontmatter         TEXT NOT NULL,                                 -- JSON snapshot of frontmatter at publish
  frontmatter_schema  TEXT,                                          -- optional JSON Schema; when set, workspace docs are validated against it on write
  published_by        TEXT REFERENCES users(id) ON DELETE SET NULL,
  published_at        INTEGER NOT NULL,
  UNIQUE(template_id, version)
);

CREATE INDEX template_versions_template_idx ON template_versions(template_id, version DESC);
```

**Notes:**
- `doc_type` constrains which document type the template produces (`'agent'` or `'trigger'`). Other doc types (work_item, page, comment, agent_run) are not templatable in this phase.
- `slug` is unique across all live templates (regardless of doc_type) — so you can't have an agent template and a trigger template both called `inbox-anonymizer`. Slug-uniqueness scoped by doc_type would be reasonable too; we pick globally-unique for clarity in tool listings.
- `current_version` is the version that workspaces installing this template land on. Stays at `0` until the first version is published.
- `template_versions.frontmatter` stores a JSON-encoded snapshot. Once published, it's immutable.

### 3b. Workspace doc → template link

The Phase 2.6 spec already adds optional `template: { id, version }` to agent and trigger frontmatter. Phase 2.7 actually wires the install/update flow that consumes it.

```ts
template: z.object({
  id: z.string().uuid(),         // templates.id
  version: z.number().int(),     // template_versions.version
}).optional()
```

Inserted at install time; updated on workspace-doc-update-to-newer-version; cleared on template delete (orphaning — see 3c).

### 3b-extra. Optional schema validation on workspace docs

When a template version declares `frontmatter_schema` (non-null JSON Schema), workspace documents installed from that version are validated against it on every PATCH that touches frontmatter. This gives templates a *contract shape*, not just a snapshot.

Behavior:
- **At install time:** the workspace doc's initial frontmatter copies from the template version's snapshot (which is valid by construction).
- **At PATCH time:** if the workspace doc has `template: { id, version }` set, `services/documents.ts::updateDocument` looks up the template version's `frontmatter_schema`. If present, validates the post-patch frontmatter against it. Reject with 422 `TEMPLATE_SCHEMA_VIOLATION` if invalid, including the validation errors in the response body.
- **At sync time:** when updating a workspace doc to a newer template version, the diff machinery includes schema changes (added/removed/changed keys with new constraints) in the conflict surface. Schema-removed keys that the workspace doc still uses → conflict (user picks "remove locally" or "block update").
- **Without a schema:** templates without `frontmatter_schema` work as today — free-form frontmatter, no contract.

Implementation:
- JSON Schema (draft 2020-12) is the format. Validated via a small dependency (`ajv` or similar; we already ship Zod so we could compile Zod → JSON Schema, but JSON Schema is the lingua franca for template authors).
- Schema is editable in the template editor (Versions tab → publish-new-version dialog includes a JSON Schema field alongside body + frontmatter).
- Validation is opt-in per template — Stefan can ship strict templates for production agents and loose templates for prototypes.

### 3c. Template delete behavior

Per the locked decision: orphan the workspace doc, clear `template` frontmatter, leave doc intact.

Mechanics:
- DELETE on a template sets `templates.deleted_at` (soft delete at the template row level).
- Same transaction: for every workspace doc with `template.id` matching, clear the `template` frontmatter (PATCH the JSON field). Emit `template.workspace_doc_orphaned` per affected doc.
- Workspace docs remain functional — they just lose the link to the template they were installed from. No more "Update available" banner. Customer keeps their work.
- `templates.deleted_at` is non-null so we don't lose audit history; the row stays in the DB. The catalog UI hides deleted templates (filtered by `WHERE deleted_at IS NULL`).
- Re-creating a template with the same slug after a delete is allowed (the unique-on-slug index has `WHERE deleted_at IS NULL`, so deleted rows don't block new ones).

### 4a. `services/templates.ts`

```ts
listTemplates(tx, { docType?, includeDeleted? }): Promise<Template[]>

getTemplate(tx, slugOrId, { includeDeleted? }): Promise<Template | null>

createTemplate(tx, {
  docType, slug, title, description?, createdByUserId
}): Promise<Template>
  // Validates slug unique. Inserts row with current_version=0.

updateTemplate(tx, id, {
  title?, description?
}): Promise<Template>
  // Only metadata fields. Body + frontmatter are immutable per-version.

publishTemplateVersion(tx, id, {
  title, body, frontmatter, publishedByUserId
}): Promise<TemplateVersion>
  // Inserts new template_versions row at version=current_version+1.
  // Updates templates.current_version atomically.
  // Returns the new version row.

deleteTemplate(tx, id): Promise<Template>
  // Sets templates.deleted_at.
  // Orphans every workspace doc carrying this template.id (clears template frontmatter).

listVersions(tx, id): Promise<TemplateVersion[]>

getVersion(tx, id, version): Promise<TemplateVersion | null>

installTemplateToWorkspace(tx, {
  templateId, workspaceId, slugOverride?, version?
}): Promise<Document>
  // Creates a new workspace agent or trigger doc.
  // Copies frontmatter + body from the chosen version (defaults to current_version).
  // Sets template: { id, version } on the new doc.
  // Auto-mints API token if doc_type='agent' (Phase 2.5 cascade).
  // Returns the new workspace doc.

updateWorkspaceDocToTemplateVersion(tx, {
  workspaceDocId, targetVersion, conflictResolutions?
}): Promise<{ document: Document, conflicts: DiffConflict[] | null }>
  // Walks the diff between installed version and target version.
  // If no conflicts: patches frontmatter + body to target version, bumps template.version frontmatter.
  // If conflicts and no resolutions: returns the doc unchanged + conflicts list.
  // If conflicts and resolutions: applies the chosen resolutions per conflict.
  // Workspace-local edits without conflict are preserved.

promoteDocToTemplate(tx, {
  workspaceDocId, slug, title, description?, groupId?, schema?, createdByUserId
}): Promise<{ template: Template, version: TemplateVersion, updatedDoc: Document }>
  // One-shot extract: takes an existing workspace agent or trigger and turns it into a template.
  // Transactional steps:
  //   1. Create a new template row (slug must be unique).
  //   2. Publish v1 with the doc's current frontmatter + body as the snapshot.
  //   3. Set the workspace doc's frontmatter.template = { id: <new>, version: 1 }.
  //   4. Auto-mint of agent token is unchanged — the original doc keeps its existing token.
  // No data is duplicated and no document deleted; the live workspace doc is just linked back to the new template.
  // Emits template.created + template.published + template.installed_to_workspace in sequence.
```

All mutations transactional via `emitEvent` — events listed in §5.

### 4b. `lib/templates/diff.ts`

Pure function. Three-way merge between (a) installed-version frontmatter+body, (b) current workspace-doc frontmatter+body, (c) target-version frontmatter+body.

```ts
type DiffResult = {
  bodyAction: 'unchanged' | 'apply_target' | 'preserve_local' | 'conflict';
  bodyDiff: { from: string; to: string; localEdit?: string };  // for UI rendering

  frontmatterChanges: Array<{
    key: string;
    installedValue: unknown;
    localValue: unknown;
    targetValue: unknown;
    action: 'unchanged' | 'apply_target' | 'preserve_local' | 'conflict';
  }>;

  hasConflicts: boolean;
};

export function diffTemplateUpdate(input: {
  installedVersion: { body: string; frontmatter: Record<string, unknown> };
  currentDoc:       { body: string; frontmatter: Record<string, unknown> };
  targetVersion:    { body: string; frontmatter: Record<string, unknown> };
}): DiffResult;
```

Resolution rules:
- For each frontmatter key:
  - `installed == local == target`: unchanged.
  - `installed == local && local != target`: target changed it, local didn't — apply target.
  - `installed != local && local == target`: local edited to the same value target chose — apply target (idempotent).
  - `installed == target && installed != local`: local edited but target didn't change — preserve local.
  - `installed != local && installed != target && local != target`: both edited differently — conflict.
- For body: same three-way logic but at the whole-string level (not line-by-line). v1 doesn't try to merge body diffs; if both the local doc and the target version changed the body relative to the installed version, it's a conflict. The UI shows both texts side-by-side and lets the user pick "keep local" or "use target."

Edge case: a frontmatter key that exists in `installed` and `target` but was deleted in `local` is treated as a conflict (deletion is an edit).

### 4c. `routes/templates.ts`

Instance-level routes (no `/w/:wslug/` prefix):

```
GET    /api/v1/templates                          -- list, optional ?doc_type=agent|trigger&group_id=...
GET    /api/v1/templates/:idOrSlug
POST   /api/v1/templates                          -- create (metadata only; no version yet)
PATCH  /api/v1/templates/:id                      -- update metadata (including template_group_id assignment)
DELETE /api/v1/templates/:id                      -- soft delete + orphan workspace docs
GET    /api/v1/templates/:id/versions
GET    /api/v1/templates/:id/versions/:version
POST   /api/v1/templates/:id/versions             -- publish a new version (body + frontmatter + optional frontmatter_schema)
POST   /api/v1/templates/:id/install              -- body: { workspace_id, slug_override?, version? }
POST   /api/v1/workspace-docs/:docId/sync-to-template-version
                                                  -- body: { target_version, conflict_resolutions? }
POST   /api/v1/workspace-docs/:docId/promote-to-template
                                                  -- body: { slug, title, description?, group_id?, schema? }

GET    /api/v1/template-groups
POST   /api/v1/template-groups                    -- body: { slug, title, description? }
PATCH  /api/v1/template-groups/:id
DELETE /api/v1/template-groups/:id                -- soft delete; templates in group keep template_group_id (set to NULL via FK ON DELETE SET NULL)
```

15 verbs total.

**Auth:**
- Session auth: the user must have an `instance_admin` flag on their user row (new column added in this phase — see 3a-extra below). Pre-existing users are not admins by default; Stefan flips the flag manually via SQL on initial deploy, then can promote others via the UI.
- Bearer auth: token must carry the `templates:admin` scope. Tokens with workspace scopes don't have this — `templates:admin` is instance-level and granted only on minting.

**Validation errors:**
- `TEMPLATE_SLUG_TAKEN` (422): create or rename with a slug already used by a live template.
- `TEMPLATE_DOC_TYPE_INVALID` (422): doc_type not `'agent'` or `'trigger'`.
- `TEMPLATE_VERSION_NOT_FOUND` (404): install with version that doesn't exist.
- `TEMPLATE_DELETED` (404): operations on a soft-deleted template (except hard-delete-by-id for cleanup).
- `WORKSPACE_DOC_NOT_TEMPLATED` (422): sync attempted on a doc with no `template` frontmatter.
- `TEMPLATE_UPDATE_CONFLICTS` (409): sync returns conflicts; response body includes the `DiffConflict[]` so the client can render the resolution UI.

### 3a-extra. `users.instance_admin` flag

Schema addition to the existing `users` table:

```sql
ALTER TABLE users ADD COLUMN instance_admin INTEGER NOT NULL DEFAULT 0;  -- 0 = false, 1 = true
```

Migration sets it `0` for all existing users. Stefan manually flips his own row via `UPDATE users SET instance_admin = 1 WHERE email = ...` on initial deploy. The `/settings/admin/users` admin UI for promoting/demoting other users is **out of scope** for Phase 2.7 — Stefan uses SQL on a fresh deploy, and we add the UI when there's actual demand (multi-admin instance).

Existing routes that already check user identity don't need changes; `instance_admin` is consulted only by the new `requireInstanceAdmin()` middleware applied to template routes.

## 5. Events

New event kinds added to `KNOWN_EVENT_KINDS`:

| Kind | Payload | Fired when |
|---|---|---|
| `template.created` | `{ template_id, doc_type, slug, group_id?, created_by }` | Template row inserted (via create_template OR promote_doc_to_template). |
| `template.updated` | `{ template_id, changed_fields }` | Template metadata patched (including group assignment). |
| `template.published` | `{ template_id, version, published_by }` | New `template_versions` row inserted. |
| `template.deleted` | `{ template_id, slug, orphaned_doc_count }` | Template soft-deleted. |
| `template.installed_to_workspace` | `{ template_id, version, workspace_id, document_id, source: 'install' \| 'promote' }` | Template installed into a workspace (via install_to_workspace OR promote_doc_to_template — `source` discriminates). |
| `template.workspace_doc_updated` | `{ template_id, document_id, from_version, to_version }` | Workspace doc synced to a newer template version. |
| `template.workspace_doc_orphaned` | `{ template_id, document_id }` | Template delete cleared a workspace doc's `template` frontmatter. Fires per affected doc inside the delete transaction. |
| `template_group.created` | `{ group_id, slug, created_by }` | Template group row inserted. |
| `template_group.updated` | `{ group_id, changed_fields }` | Template group metadata patched. |
| `template_group.deleted` | `{ group_id, slug, detached_template_count }` | Template group soft-deleted; templates' `template_group_id` set to NULL via FK. |

All events flow through the standard event bus + SSE (using existing `/api/v1/w/:wslug/events`? No — templates are instance-level, so we need a new SSE endpoint for instance-scoped events; see §5a).

### 5a. Instance-scoped SSE

The existing SSE endpoint is workspace-scoped (`/api/v1/w/:wslug/events`). Template events don't belong to any workspace; they're instance-level. New endpoint:

```
GET /api/v1/events?kinds=template.*
```

Auth: session (must be `instance_admin`) OR bearer with `templates:admin`. Workspace tokens reject with 403.

Filters: `?kinds=` works the same way. No `?project=`, `?parent=`, `?agent=` (those don't apply at instance scope). `Last-Event-Id` replay works.

For convenience, **`template.installed_to_workspace`, `template.workspace_doc_updated`, `template.workspace_doc_orphaned` also flow through the workspace SSE** for the affected workspace, since they describe a workspace-doc change. The other four (`template.created/updated/published/deleted`) only flow through the instance SSE because they don't touch a specific workspace.

This dual-flow is the only place an event lands on two SSE channels. The event row in the `events` table itself stays single — both channels read from the same source.

### MCP_TOOL_GROUPS — new `INSTANCE_ADMIN` group

`packages/shared/src/mcp-tool-groups.ts` (existing from Phase 2.5) gains a new group:

```ts
export const MCP_TOOL_GROUPS = {
  READ:           [...],     // existing
  WRITE:          [...],     // existing
  DELETE:         [...],     // existing
  AGENTS_WRITE:   [...],     // Phase 2.6
  INSTANCE_ADMIN: [
    'list_templates',
    'get_template',
    'create_template',
    'update_template',
    'publish_template_version',
    'delete_template',
    'install_template_to_workspace',
    'update_workspace_doc_to_template_version',
    'promote_doc_to_template',
    'list_template_groups',
    'create_template_group',
    'update_template_group',
    'delete_template_group',
  ],
} as const;
```

The agent-editor tools picker (`ToolsField` component from Phase 2.5) filters the `INSTANCE_ADMIN` group out by default. It's only shown when **the editing context** has `templates:admin` — i.e. either an instance-admin session is authoring the agent, or a meta-agent with `templates:admin` is creating another agent via MCP `create_agent`. Default workspace agents authored through the workspace popover never see these tools in their picker.

Runtime enforcement is still on the bearer token at call time (a misconfigured agent with `tools: ['list_templates']` but no `templates:admin` scope gets `-32602`), but UI filtering prevents the misconfiguration from happening in the first place.

`templates:admin` is the only scope that grants these tools. It's a **new instance-scope** added to the existing scopes vocabulary. Tokens with workspace-class scopes (`documents:write`, `agents:write`, etc.) do not implicitly have `templates:admin`.

`get_agent_self` (from Phase 2.6) returns the calling agent's full document, which includes `template` frontmatter if installed — so an agent can introspect whether it was installed from a template and which version.

A new tool `update_self_to_template_version` is NOT shipped — agents can't update themselves through a "self" tool. If a meta-agent wants to update another agent (including potentially itself via id), it goes through the regular `update_workspace_doc_to_template_version` tool. That keeps the surface uniform.

### 7b. Template editor (slideover)

Reuses the existing slideover shell from Phase 2.6 (Fields · Activity · Runs for agents / triggers). For templates, the tab set is **Fields · Versions · Activity · Workspaces**:

- **Fields:** the template's current editable metadata + body. Editing here doesn't change any published version — it's a working draft. Click "Publish v(N+1)" to snapshot the current state into a new immutable version row.
- **Versions:** lists every `template_versions` row. Click to view a frozen snapshot. Side-by-side diff between any two versions available via a "Compare" toggle (uses the same diff machinery as `lib/templates/diff.ts`).
- **Activity:** standard `ActivityPanel`. Reads events with `documentId = template.id` — `template.created`, `template.updated`, `template.published`, `template.installed_to_workspace`, etc.
- **Workspaces:** shows which workspaces have an instance of this template and which version they're pinned to. Click a row → opens the workspace doc's slideover stacked on top.

Header right side:
- `[Publish v(N+1)]` button — always visible when there are unpublished changes to title/body/frontmatter (computed against `current_version`'s snapshot).
- `⋯` menu — Delete (confirm dialog showing the workspace count that will be orphaned).

There is no Runs tab on a template editor — templates don't run, their workspace instances do. The Workspaces tab is the equivalent navigation surface.

## 10. Open questions (for the implementation plan)

- **Conflict resolution UX for many simultaneous conflicts.** v1 has a vertical list of conflict rows; if a template update produces 20 conflicts, the dialog is unwieldy. Probably acceptable for v1 — large updates are operator-pain anyway and Stefan reviews them.
- **Workspace-doc rename on install.** If a workspace already has an agent at slug `reply-drafter` and tries to install a template with the same slug, the install needs to either reject (slug collision) or generate a unique slug (`reply-drafter-2`). Recommendation: reject; require `slug_override` param. Surfaced clearly in the install UI.
- **Frontmatter key removal across versions.** If v1 has key `foo` and v2 removes it, the diff machinery treats removal as a change. Need to decide: does sync remove the key from the workspace doc, or preserve it? Recommendation: preserve (the workspace doc carries data; we don't drop data silently). Document this in the diff rules.
- **`instance_admin` granularity.** v1 is one boolean. Future: per-resource admin roles (`can_manage_templates`, `can_promote_users`, etc.). Out of scope for 2.7.
- **Audit of who-installed-what.** `template.installed_to_workspace` event captures it. UI to surface install history is Phase 7+ polish.

## Appendix B: Scope summary across phases

After Phase 2.7 ships, the workspace + instance scope vocabulary is:

| Scope | Level | Granted to |
|---|---|---|
| `documents:read|write|delete` | Workspace | Default agent + workspace bearer tokens |
| `fields:write` | Workspace | Workspace bearer tokens |
| `views:write` | Workspace | Workspace bearer tokens |
| `tables:write` | Workspace | Workspace bearer tokens |
| `statuses:write` | Workspace | Workspace bearer tokens |
| `agents:write` | Workspace | Meta-agents only (Phase 2.6) |
| **`templates:admin`** | **Instance** | **Instance-admin tokens only (Phase 2.7)** |

`templates:admin` is the first instance-level scope. Future instance scopes (e.g. `users:admin` for promoting other admins) follow the same pattern.
