# Phase 3.x — Model-initiated approval (human-in-the-loop completion) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: `netdust-core:ntdst-execute-with-tests` (wraps `superpowers:subagent-driven-development`). Step 2.5 plan-freshness per task; two-stage review per task.

**Goal:** Close Folio's two implementable gaps against HumanLayer's 12-Factor Agents (Factor 7 "contact humans with tools", Factor 9 "escalate not just fail") by giving the agent a way to **pause itself for human approval** — both when the agent *judges* an action high-stakes (a `request_approval` tool) and when the D-9 error sub-cap trips (escalate instead of dead-end). The RESUME side already exists (D-5); this builds the missing PAUSE/REQUEST side. **Factor 3 ("hide resolved errors") was DROPPED from this plan after pressure-test — see "Pressure-test findings" below: it's not implementable without provider-API 400s under Folio's message model.**

> **⚠️ PRESSURE-TESTED 2026-05-30 (adversarial design review, BEFORE any code).** The first draft had 3 BLOCKERs (Factor-3 pruning impossible, the plan comment re-entered the reactor plane, role-attribution inversion on resume) + several IMPORTANTs. ALL are fixed in this revision. The "Pressure-test findings + resolutions" section below records each. The original self-review section rubber-stamped the broken items — do NOT trust a plan's self-review over an adversarial pass.

**Architecture:** The runner already has the rails — the `awaiting_approval` state, the `kind=plan`/`kind=approval` comment convention, `buildResumeMessages`, the D-5 `resume_run`/`reject_run` triggers, and the poller's `resume_of` routing. What's missing is the path that PUTS a run into `awaiting_approval` from inside `runLoop`:
1. A `request_approval` tool the agent can call → the loop posts a `kind=plan` comment (the proposed action + rationale), transitions `running → awaiting_approval`, and returns (pauses). The human's `kind=approval`/`kind=rejection` comment then drives D-5's existing resume/reject.
2. The D-9 escalation variant: when `consecutiveToolErrorRounds` hits the cap, instead of `failRun(tool_error)`, optionally ESCALATE — post a `kind=plan` comment summarizing the stuck state and transition `awaiting_approval` (gated on the agent's `requires_approval` or a new flag), so a human can intervene rather than the run dead-ending.

**Tech Stack:** the hand-rolled `runLoop` on `provider.stream()`, `executeTool`/`agent-tools.ts` registry, `agent-run-schema.ts` state machine, `services/comments.ts`, the existing D-5 trigger wiring.

**Status:** PLAN ONLY — written 2026-05-30. Build AFTER Phase 3 Sub-phases E + F (do NOT interrupt the E web-UI work). Originating analysis: `memory/project_12-factor-agents-audit.md`. The 12-factor doc: github.com/humanlayer/12-factor-agents (Factors 7, 8, 9, 3).

## Why this is one feature, not three

The three gaps compose:
- **Factor 7** (model requests human via a tool) needs a `request_approval` tool + the `running → awaiting_approval` transition.
- **Factor 9 escalation** (after N errors, escalate to a human instead of failing) REUSES that same transition — it's "the runner calls request_approval on the model's behalf when stuck."
- **Factor 3** (hide resolved errors) is a small context-build refinement that pairs naturally: once we're managing approval context, we also prune resolved tool-errors from history.

The resume/reject machinery (D-5) is DONE, so this is mostly the write/pause side + one context tweak.

## Ground-truth (verified against live source 2026-05-30, HEAD `255c3e1`)

- **State machine** (`agent-run-schema.ts:119`): `running: ['completed', 'failed']` — **does NOT currently allow `running → awaiting_approval`.** This plan ADDS it. `awaiting_approval: ['running', 'rejected', 'failed']` (resume/reject paths exist). `planning: ['awaiting_approval', 'running', 'failed']`.
- **`buildResumeMessages`** (`runner.ts:496`) already assembles approval context from `listComments({parentId, kind:['plan','approval']})` — so a `kind=plan` comment the agent posts at pause time is ALREADY picked up on resume. The rendezvous point is `kind=plan`.
- **`postAgentComment`** (`runner.ts:820`) currently accepts ONLY `kind: 'result' | 'comment'`. **Must extend to accept `'plan'`** (the comment-schema already has a `plan` kind — verify `comment-schema.ts`).
- **`buildToolDefs`** (`runner.ts:524`) advertises only the agent's `frontmatter.tools` (a closed `V1_MCP_TOOLS` enum, `agent-schema.ts:10`). `request_approval` must be advertised ALWAYS (not gated on the agent's tool list) — it's a control primitive, not a workspace tool. ALSO: `buildToolDefs` carries a STALE comment ("C-8 skeleton... D-3 wires real schemas") — D-3 shipped; advertise the real `request_approval` input schema, and note the staleness for a separate cleanup.
- **`requires_approval: boolean`** (`agent-schema.ts:19`, default false) ALREADY EXISTS on agent frontmatter but is currently UNUSED by the runner. This plan wires it: `requires_approval: true` = config-driven mandatory pause-before-first-action (the conservative model); the `request_approval` tool = model-driven pause-when-it-judges (the flexible model). Both reach the same `awaiting_approval` state.
- **D-9 escalation hook** (`runner.ts:~704`): the `consecutiveToolErrorRounds >= MAX_CONSECUTIVE_TOOL_ERRORS → failRun(tool_error)` block is where the escalation variant branches.
- **`executeTool`** dispatches registered tools; `request_approval` registers in `agent-tools.ts`/`agent-tools-registry.ts` like the others BUT its "execution" is special — the runner must intercept it (it can't be a normal handler that does DB work, because the runner needs to break the loop). Two design options below.

## Design decision (resolve at build, surface at approval)

**How does `request_approval` break the loop?** A normal tool runs via `executeTool` and returns a result string the loop feeds back. But `request_approval` must PAUSE, not return-and-continue. Two options:

- **Option A — runner-intercepted control tool (RECOMMENDED).** `request_approval` is advertised to the provider (via `buildToolDefs`) but the runner INTERCEPTS it before `executeTool`: when a collected tool_call has `name === 'request_approval'`, the runner does NOT dispatch it — instead it reads the call's args (`{summary, proposed_action}`), posts a `kind=plan` comment with that content, transitions `running → awaiting_approval`, and returns (pause). This keeps the pause logic in the runner (Factor 8 — own your control flow) and avoids a tool handler that needs to reach back into run state. The model SEES `request_approval` in its tool list and calls it like any tool; the runner gives it special meaning.
- **Option B — registered tool that throws a sentinel.** `request_approval` is a real `executeTool` handler that throws a typed `ApprovalRequested` sentinel the runner catches. More uniform but smuggles control flow through the error path (confusing; collides with D-9's error handling). REJECT unless A proves awkward.

**Lock Option A.** Rationale: the runner already special-cases the cancel check + budget check inline in the loop; `request_approval` is the same shape of control primitive.

## Threat model — Phase 3.x extension (mitigations 67–70)

> Added 2026-05-30 at plan-write. EXTENDS B(1-22)+C(23-47)+C.3(48-53)+D(54-63)+D-9(64-66). This feature lets an agent PAUSE itself and surfaces a `kind=plan` comment (model-authored content) to humans, and adds a `running → awaiting_approval` transition reachable from the loop. Inherited mitigations (esp. 28 sanitization, 43 approval-race, the autonomy gate) remain in force.

### What we're defending
- **The approval pause as a NEW model-reachable state transition.** A prompt-injected agent must not be able to use `request_approval` to escape resource caps (e.g. pause to reset the token budget / round cap) or to spam humans with approval requests.
- **The `kind=plan` comment as model-authored content shown to a human + fed back on resume.** It must be sanitized like any agent-authored comment (no smuggled instructions that a careless human auto-approves; no injection into the resume context that escalates privilege).

### Attacks → mitigations
67. **Approval-loop resource-cap reset.** Agent calls `request_approval` repeatedly to pause/resume and dodge the per-run token budget or `MAX_TOOL_ROUNDS` (a fresh resume run might reset counters). → **Mitigation 67:** a resume run INHERITS the original's `chain_id` (already true, D-5) and the chain-level token cap (`FOLIO_MAX_CHAIN_TOKENS`, mitigation 30) + fanout cap bound the whole approval chain, not just one run. ADD: a per-chain approval-request cap (e.g. an agent can't request approval more than N times in a chain) OR rely on the chain caps — decide at build. Test: an agent that loops request_approval is bounded by the chain token/fanout cap, not unbounded.
68. **Approval-request spam / human-fatigue DoS.** Agent floods a parent with `kind=plan` comments. → **Mitigation 68:** `request_approval` transitions `running → awaiting_approval` and RETURNS (the run pauses — it can't post a second plan in the same run). A new plan requires a full resume cycle (human approval → new run). So one run = at most one approval request. Test: a run that calls request_approval twice in one round — only the FIRST takes effect (the loop returns after the first); assert one `kind=plan` comment + one transition.
69. **Model-authored `kind=plan` content injection.** The `summary`/`proposed_action` args are model-generated (and the model may be prompt-injected). They're shown to a human (who might auto-approve) and fed back into the resume context. → **Mitigation 69:** the `kind=plan` comment body is model content — treat it like any agent comment (it already flows through `createComment`'s author resolution + the comment renders as agent-authored, clearly attributed). Do NOT auto-approve based on plan content. The human approval (mitigation 42 — any commenter with `comments:write`) is the gate. On resume, the plan is `role:'user'` context (buildResumeMessages) — same trust level as any parent comment. NO new privilege. Document that plan content is untrusted-model-output shown for human judgment, not executed.
70. **Escalation (Factor 9) bypasses the error cap / human-fatigue ping-pong.** If D-9's error-cap escalation transitions to `awaiting_approval` instead of failing, a stuck agent could be resumed and re-stick, looping — bounded by chain caps (fanout 25 / tokens 200k) but a human could approve ~25 times before an inevitable failure. → **Mitigation 70:** escalation is (a) GATED — only when `escalate_on_error` is set; default = fail as today (D-9 unchanged); AND (b) attempt-CAPPED per chain — a per-chain escalation counter (default `MAX_CHAIN_ESCALATIONS = 2`): after N escalations in a chain, `failRun(tool_error)` instead of escalating again (no infinite human-toil). Test: escalation OFF → `tool_error` fail (D-9 unchanged); escalation ON, first cap-hit → `awaiting_approval` + `kind=plan` stuck-state summary; escalation ON, (N+1)th cap-hit in the same chain → `failRun(tool_error)` (the attempt cap fires).

## Pressure-test findings + resolutions (2026-05-30 adversarial review)

Each finding, why it broke the first draft, and how this revision resolves it. Verified against live source (anthropic.ts tool_use/tool_result pairing, trigger-matcher.ts react loop, buildResumeMessages, recoverOrphanRuns).

- **BLOCKER (was Factor-3 pruning) — DROPPED.** "Prune resolved `{role:'tool'}` error messages after a success" is NOT implementable: errors + successes from the SAME round share ONE `assistant` message carrying all `tool_calls`; deleting a tool-RESULT orphans its `tool_use` block → Anthropic `400 invalid_request` (every `tool_use` id must have a matching `tool_result` in the next turn — anthropic.ts:21-33). Removed from this plan entirely. (If ever revisited: the ONLY balanced operation is pruning whole rounds — assistant + all its results together — when an entire round was recoverable-error and a later round succeeded. Out of scope here.)
- **BLOCKER — `kind=plan` comment re-enters the reactor plane.** `postAgentComment`→`createComment` ALWAYS emits `comment.created` (comments.ts:391-407); the dispatcher fans it to the trigger-matcher (kinds include `comment.created`). An agent-authored plan comment is `isAgentOriginated` (actor `agent:<slug>`) AND could fire any user-authored `{on_event: comment.created}` trigger → self-loop / suppressed-signal spray. **Resolution (Task 3x-2 Step 0):** add a guard in `triggerMatcher.react` (or `maybeCreateRun`) that SKIPS events where `payload.kind === 'plan'` — agent plan-requests are control signals, not trigger fuel. Pin with a test (a plan comment fires ZERO runs + ZERO suppressed events).
- **BLOCKER — role-attribution inversion on resume.** `buildResumeMessages` pushes BOTH `kind=plan` and `kind=approval` comments as `role:'user'` (runner.ts:513-518). So a resumed agent sees its OWN proposed plan replayed in the HUMAN's voice. The first draft falsely asserted "coherent." **Resolution (Task 3x-2 Step 3a):** in the buildResumeMessages plan/approval loop, render a comment authored by the run's own agent (`author === agent:<self>` / `kind==='plan'`) as `role:'assistant'` ("I proposed: …"); keep human approval/rejection as `role:'user'`. Test: a resume after request_approval produces an assistant turn for the plan + a user turn for the approval (correct attribution).
- **IMPORTANT — side-effects before pause.** The loop collects ALL round tool_calls then executes them in a BATCH (runner.ts:604, 640-690). If `request_approval` co-occurs with `write_document`, the write executes BEFORE the pause. **Resolution (Task 3x-2 Step 3, locked):** PRE-SCAN `collectedToolCalls` BEFORE the execution loop — if ANY call is `request_approval`, pause IMMEDIATELY and execute NONE of the sibling calls. The tool description tells the model to call `request_approval` ALONE. Test: a round with `write_document` + `request_approval` → the document is NOT written, run pauses.
- **IMPORTANT — `requires_approval` only guards the FIRST action (spec mismatch).** The first draft sold it as "nothing happens without sign-off"; as designed it gates only the first run's first action (resume runs skip it via `resume_of`). **Resolution:** downscope the DESCRIPTION to the honest contract — `requires_approval: true` = "the agent must propose-and-pause before its FIRST action; the human's approval authorizes the run to proceed." (Re-arming on every action = a separate future 'approve-every-step' mode; out of scope.) Detection signal is `ctx.fm.resume_of != null` (NOT buildResumeMessages — that returns a Message[] with no resume marker). Stated explicitly in Task 3x-3.
- **IMPORTANT — escalation human-fatigue ping-pong.** Escalate→approve→resume→re-stick→escalate is bounded by chain caps (fanout 25 / tokens 200k) but a human could approve ~25 times before an inevitable failure. **Resolution (Task 3x-4):** add a per-chain escalation attempt cap (default 2) — after N escalations in a chain, `failRun(tool_error)` instead of escalating again. Promoted from open-question to a required mitigation (70).
- **MINOR (M-1) — `kind=plan` comment has no `run_id` linkage** that E-6's approval-buttons need (`useRun(comment.frontmatter.run_id)`). `createComment` has no `run_id` passthrough today. **Resolution:** E-6 resolves the run via `(parent, agent, status=awaiting_approval)` (the `getPendingApprovalRun` shape) rather than a comment→run_id link — OR a future createComment `run_id` field. Noted for the E-6 builder; not a 3.x blocker.
- **MINOR (M-2) — the registered `request_approval` throw-handler would be classified RECOVERABLE** (not fatal) by D-9's `isFatalToolError` (it doesn't start with `forbidden: scope`/`method not found`), so a stray non-intercepted call would be fed back + retried, not fail loudly. **Resolution (Task 3x-2 Step 2):** add `request_approval` "must be runner-intercepted" to `isFatalToolError`'s fatal set, OR do not register a handler at all (so an un-intercepted call hits `method not found` → fatal). Prefer the latter (simpler) — `request_approval` is advertised via buildToolDefs but NOT registered in the executeTool registry; interception is the only path.
- **MINOR (M-3) — orphan-recovery interaction.** A long-paused `awaiting_approval` run has `worker_started_at` set; verify `recoverOrphanRuns` only reclaims `status='running'` rows (it should — agent-runs.ts) so a paused run isn't failed as a crashed worker. **Resolution (Task 3x-2 Step 3):** pin with a test — seed an `awaiting_approval` run with a stale `worker_started_at`, run recovery, assert it stays `awaiting_approval`.
- **MINOR (M-4) — paused run must not also post `kind=result`.** The interception `return`s before `postResultAndComplete`. Test assertion: a paused run has exactly one `kind=plan` comment and ZERO `kind=result`.
