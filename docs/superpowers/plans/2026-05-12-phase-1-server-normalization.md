# Phase 1 Server Normalization — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reconcile server response shapes, schema, and route names with the canonical spec (`docs/superpowers/specs/2026-05-11-phase-1-core-crud-design.md`) and `docs/FOLIO-BRIEFING.md` so the Phase 1 frontend plan can resume at Task 5 without per-call shape patches.

**Architecture:** Pure normalization — no new features, no behavior changes the client doesn't already expect. Three classes of fix: (a) Drizzle schema columns the spec implies but the schema never grew (`workspaces.updatedAt`, `projects.description`/`archivedAt`/`updatedAt`); (b) detail-endpoint response shape — drop the inner `{ resource: row }` wrap so post-envelope-unwrap the client sees the bare row, as the spec already says (§4 detail endpoints return `{ data: Document }`); (c) magic-link route rename from the shortened `/magic/*` to the canonical `/magic-link/*` per FOLIO-BRIEFING.md §8. The workspaces list shape stays `[{ workspace, role }]` per the user's decision — we update the spec to match the server, not the other way around. The Workspace client type also drops its premature AI fields (those belong on the settings endpoint per Phase 0 §46).

**Tech Stack:** Bun + Hono + Drizzle + SQLite (server). React + TanStack Query + Vitest (client). No new dependencies.

**Spec:** `docs/superpowers/specs/2026-05-11-phase-1-core-crud-design.md` §4 (API surface, response envelope). The four mismatches between the spec and the shipped server are inventoried in memory entry `phase-1-server-spec-mismatch`.

**Frontend plan it unblocks:** `docs/superpowers/plans/2026-05-11-phase-1-frontend.md` — Tasks 1-4 are committed; Task 5 (root auth gate + Toaster) is the resume point. This normalization plan runs **first**, then Task 4 hooks get re-aligned (Task 9 below), then frontend plan resumes at Task 5.

## Task 2: Unwrap workspaces detail responses

**Files:**
- Modify: `apps/server/src/routes/workspaces.ts`

**Why:** Spec §4 says detail endpoints return `{ data: Workspace }`. The route currently wraps as `{ data: { workspace: row } }`. After the client envelope unwrap (Task 2 of the frontend plan), components see `{ workspace: ... }` instead of the bare row. Drop the inner wrap on POST, GET item, PATCH item. Collection `GET /` stays as-is (the wrapped `{ workspace, role }` is the agreed shape — see Task 5).

- [ ] **Step 1: Edit `workspaces.ts` POST response (line 72)**

Find:

```ts
    return jsonOk(c, { workspace: { id, slug, name } }, 201);
```

Replace with:

```ts
    return jsonOk(c, { id, slug, name }, 201);
```

- [ ] **Step 2: Edit `workspaces.ts` GET item response (line 81)**

Find:

```ts
item.get('/', (c) => jsonOk(c, { workspace: getWorkspace(c), role: getRole(c) }));
```

Replace with:

```ts
item.get('/', (c) => jsonOk(c, { ...getWorkspace(c), role: getRole(c) }));
```

Rationale: detail endpoint returns the bare workspace row. Role is included as a sibling key on the same object (not a separate level) — matches the spirit of the collection shape but flattens at the item level since the item is a single resource, not a membership row.

- [ ] **Step 3: Edit `workspaces.ts` PATCH response (line 97) — also set updatedAt**

Find:

```ts
  await db.transaction(async (tx) => {
    await tx.update(workspaces).set({ name }).where(eq(workspaces.id, ws.id));
    await emitEvent(tx, {
      workspaceId: ws.id,
      kind: 'workspace.updated',
      actor: user.id,
      payload: { changes: ['name'] },
    });
  });
  return jsonOk(c, { workspace: { ...ws, name } });
```

Replace with:

```ts
  const now = new Date();
  await db.transaction(async (tx) => {
    await tx.update(workspaces).set({ name, updatedAt: now }).where(eq(workspaces.id, ws.id));
    await emitEvent(tx, {
      workspaceId: ws.id,
      kind: 'workspace.updated',
      actor: user.id,
      payload: { changes: ['name'] },
    });
  });
  return jsonOk(c, { ...ws, name, updatedAt: now });
```

Rationale: spec implies `updatedAt` advances on write; the new schema column has a `DEFAULT` but Drizzle won't trigger that on UPDATE — we must set it explicitly. Returning the new value avoids the client refetching just for the timestamp.

  createdAt: string;

    queryKey: settingsKeys.aiKeys(workspaceId),
    queryFn: () =>
      client.get<{ keys: AiKey[] }>(`/api/v1/settings/${workspaceId}/ai-keys`),
    staleTime: 60_000,

}

}
