# Phase 1.8 — Time-based + Gallery Views Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add two new saved-view types — a **time-based view** (items positioned on a date axis, with a user-selectable date field) and a **gallery view** (responsive card grid with an optional cover image) — on top of the existing list/kanban/table scaffolding.

**Architecture:** Both views inherit the proven shared scaffolding (`useDocuments`, URL-driven filters/sort, `useUpdateDocument` optimistic writes, the `DocumentSlideover`, `useFields`). The new surface area is: (1) a **view-type registry** so adding the 3rd/4th view type is one map entry, not bespoke route wiring; (2) a `dateField` column on `views`, exactly paralleling kanban's `groupBy`, picked via a popover that mirrors `BoardControls`; (3) a `cover_url` frontmatter convention + a hardened `<img>` render for the gallery. No new tables — all item data stays in `documents.frontmatter`.

**Tech Stack:** Bun + Hono + Drizzle + SQLite (server); React + Vite + TanStack Router + Tailwind + shadcn/ui + dnd-kit (web); Vitest + React Testing Library (web tests), `bun test` (server tests).

## Threat model

> This threat model covers Phase 1.8's two new view types, written 2026-06-01 at plan time. The security-relevant surface is narrow but real: the gallery view renders a **user-supplied URL** (`cover_url`) as an `<img src>` in *other workspace members'* browsers, and both views add a new view-config field (`dateField`) validated at the API boundary. The server never fetches `cover_url` — the browser does — so this is a client-side-render / stored-injection surface, not a server SSRF surface. This section is the convergence target: `/code-review` verifies against the named mitigations rather than re-discovering the surface.

### What we're defending

- **The viewing user's browser session** — a gallery card renders `frontmatter.cover_url` from a document any workspace writer (human, agent, or token) authored; a malicious URL renders in a *different* member's browser.
- **Workspace member privacy / network position** — an `<img src>` to an attacker-controlled host leaks the viewer's IP, User-Agent, and a load-timestamp (tracking-pixel class), and could target the viewer's intranet (`http://192.168.x.x/...`) from inside their browser.
- **View-config integrity** — `views.dateField` and `views.type` are persisted config; a bad value should be rejected at write time, not blow up the render for every user of that view.
- **Workspace/project isolation** — both views fetch through the existing `useDocuments` scope predicates; the new code must not introduce a query path that bypasses the workspace/project/table scope.

### Who we're defending against

- **Authenticated workspace writer (human) supplying a malicious `cover_url`** — IN scope. Has legitimate write access to documents but should not be able to inject active content or exfiltrate other viewers' data via a cover image.
- **Compromised / prompt-injected agent writing `cover_url`** — IN scope. Agents write frontmatter (Phase 2-3); a steered agent could plant a tracking/`javascript:` URL. Same mitigation as the human case (render-time sanitization), so defended by construction.
- **External attacker with no account** — OUT of scope for this surface (they can't write a document's frontmatter without a writer credential; covered by existing auth).
- **Insider with stolen credentials** — OUT of scope (acknowledged; existing auth boundary, not this feature's job).

### Attacks to defend against

1. **Active-content injection via `cover_url`**: an attacker sets `cover_url` to `javascript:...`, `data:text/html,...`, or `vbscript:...`. If rendered into an `<img src>` (or worse, interpolated into other markup) it could execute in the viewer's context.
2. **Tracking-pixel / network-position leak via `cover_url`**: an attacker sets `cover_url` to `http://attacker.example/track.gif?u=victim` or an intranet address; the viewer's browser auto-fetches it on render, leaking IP/UA/timing and hitting hosts reachable only from the viewer's network.
3. **Stored-injection broadening**: the same `cover_url` string later gets reused somewhere it's NOT in an `src` attribute (e.g. a CSS `background-image:url(...)`, an anchor `href`, a copy-as-MD export) where the sanitization assumptions differ and `)`/`"` break out of context.
4. **`dateField` / `type` config poisoning**: a write supplies a `dateField` that isn't a string, or a `type` outside the allowed enum, causing render crashes or persisting an unusable view.
5. **Scope bypass in the new views**: a new fetch/query path in the time or gallery view omits a workspace/project/table predicate and reads documents the caller shouldn't see.

### Mitigations required

1. **One shared URL sanitizer for cover images.** Add `apps/web/src/lib/safe-image-url.ts` exporting `safeImageUrl(value: unknown): string | null`. It returns the URL **only** if it is a string that parses via `new URL()` and has protocol exactly `https:` (or `http:` — decide per item below) — `javascript:`, `data:`, `vbscript:`, `file:`, blank, and unparseable values all return `null`. The gallery card calls this; when it returns `null`, the card renders the text-only fallback (no `<img>` at all). `https`-only by default; allow `http` only if a follow-up explicitly opts in. This is the single render-time gate — there is no path that renders `cover_url` into an `<img>` without passing through `safeImageUrl`.
2. **No auto-fetch of un-sanitized hosts + lazy/decoupled load.** Because mitigation 1 blocks non-`http(s)` schemes and the image only loads after passing the sanitizer, the tracking-pixel risk is reduced to "the viewer loads an https image the author chose" — an accepted residual (same as any user-pasted image link). Set `loading="lazy"` and `referrerPolicy="no-referrer"` on the gallery `<img>` so cross-origin cover loads don't leak the Folio referrer URL.
3. **`cover_url` is consumed ONLY via `safeImageUrl` and ONLY as an `<img src>`.** Do not interpolate it into `style`, `background-image`, `href`, or template strings anywhere. Add a code comment at the `safeImageUrl` definition naming this constraint, and grep all consumers during review (mitigation check below). Copy-as-MD export emits the raw frontmatter value as-is (markdown image syntax) — that's data, not active content, and is the user's own export; not broadened here.
4. **`dateField` validated as nullable string; `type` validated against the extended enum** in `apps/server/src/routes/views.ts` `baseSchema` (Zod). `dateField` defaults to `null` (meaning `due_date`) and is stored as a plain text column. The render coerces a missing/`null` `dateField` to `due_date`; an item whose value at that key isn't an ISO date simply lands in the Unscheduled tray (no throw).
5. **New views reuse `useDocuments` unchanged.** The time and gallery views MUST call the existing `useDocuments(wslug, pslug, listParams)` hook (which carries the workspace/project/table scope predicates) and MUST NOT introduce a new fetch. No raw `fetch`/query in either view component. Verified by grep at review.

### Out of scope (explicit deferrals)

- **`http://` cover URLs** — default to `https`-only; mixed-content + intranet-leak risk for `http` is acceptable-to-defer, and re-enabling is a one-line allow-list change if a customer needs it.
- **Server-side validation/fetch of `cover_url`** — the server never fetches it; we accept that a viewer's browser loads an author-chosen https image (same trust model as any pasted link). No SSRF surface because no server fetch.
- **Subresource Integrity / CSP `img-src` allow-listing** — a global CSP `img-src` policy is a platform-wide concern, not this view's; deferred to a dedicated CSP pass.
- **Removing tracking-pixel capability entirely** (e.g. proxying cover images through the server) — v1.1+; would reintroduce an SSRF surface and asset-storage cost this plan deliberately avoids.

### How to use this section

- **Controller pre-flight:** verify Task 6 (gallery card) routes every `cover_url` render through `safeImageUrl`, and Task 2 (views schema/route) validates `dateField` + extends the `type` enum, before dispatching those tasks' review.
- **`/code-review` invocations:** include "Verify code against the threat model in the plan. Each numbered mitigation should be checked: (1) one shared `safeImageUrl`, https-only, null-on-reject; (2) lazy + no-referrer on the cover `<img>`; (3) `cover_url` consumed only via `safeImageUrl` as `<img src>` — grep for any `style`/`background`/`href`/template-string use; (4) `dateField` + `type` Zod validation; (5) no new fetch in either view. Report which are in place, missing, or out-of-scope per deferrals."
- **`/evaluate` retros:** list any missing mitigation as a plan-correction defect.
- **Downstream:** the "This Week" dashboard plan and any future image-rendering surface cross-reference mitigation 1 (`safeImageUrl`) rather than re-implementing URL sanitization.

```ts
  gallery: { kind: 'gallery', label: 'Gallery', icon: Image, groupable: false, dated: false },
```

```ts
// apps/web/src/components/timeline/timeline-grouping.ts
import type { DocumentSummary } from '@/lib/api/documents';

const ISO_DATE_RE = /^\d{4}-\d{2}-\d{2}(T.*)?$/;

export interface PositionedItem {
  doc: DocumentSummary;
  start: string; // ISO date
  end: string;   // ISO date (== start for a point)
}

export interface TimelineResult {
  positioned: PositionedItem[];
  unscheduled: DocumentSummary[];
}

function isoOrNull(v: unknown): string | null {
  return typeof v === 'string' && ISO_DATE_RE.test(v) ? v : null;
}

/** Resolve the positioning date(s) for a doc. Span when start_date+end_date both valid; else the selected field as a point. */
export function buildTimeline({
  docs,
  dateField,
}: {
  docs: DocumentSummary[];
  dateField: string | null;
}): TimelineResult {
  const key = dateField || 'due_date';
  const positioned: PositionedItem[] = [];
  const unscheduled: DocumentSummary[] = [];

  for (const doc of docs) {
    const fm = doc.frontmatter as Record<string, unknown>;
    const start = isoOrNull(fm.start_date);
    const end = isoOrNull(fm.end_date);
    if (start && end) {
      positioned.push({ doc, start, end });
      continue;
    }
    const point = isoOrNull(fm[key]);
    if (point) {
      positioned.push({ doc, start: point, end: point });
    } else {
      unscheduled.push(doc);
    }
  }
  return { positioned, unscheduled };
}
```

```tsx
// apps/web/src/components/timeline/timeline-controls.tsx
import { useState } from 'react';
import { ChevronDown } from 'lucide-react';
import { Popover, PopoverContent, PopoverTrigger } from '@/components/ui/popover';
import { Icon } from '@/components/ui/icon';
import type { Field } from '@/lib/api/fields';

const triggerClass =
  'flex items-center gap-1 rounded-md px-2 py-1 text-sm hover:bg-bg-2';
const itemClass =
  'flex w-full items-center rounded px-2 py-1.5 text-left text-sm hover:bg-bg-2';

function labelFor(key: string | null, fields: Field[]): string {
  if (!key || key === 'due_date') return 'Due date';
  return fields.find((f) => f.key === key)?.label ?? key;
}

export function TimelineDateFieldPicker({
  dateField,
  fields,
  onChange,
}: {
  dateField: string | null;
  fields: Field[];
  onChange: (key: string) => void;
}) {
  const [open, setOpen] = useState(false);
  const dateFields = fields.filter((f) => f.type === 'date' || f.type === 'datetime');
  const pick = (key: string) => {
    setOpen(false);
    onChange(key);
  };
  return (
    <Popover open={open} onOpenChange={setOpen}>
      <PopoverTrigger asChild>
        <button type="button" className={triggerClass}>
          <span className="text-fg-3">Date:</span>
          <span>{labelFor(dateField, fields)}</span>
          <Icon icon={ChevronDown} size={14} />
        </button>
      </PopoverTrigger>
      <PopoverContent align="start" className="w-[200px] p-1">
        <button type="button" role="menuitem" className={itemClass} onClick={() => pick('due_date')}>
          Due date
        </button>
        {dateFields
          .filter((f) => f.key !== 'due_date')
          .map((f) => (
            <button
              key={f.key}
              type="button"
              role="menuitem"
              className={itemClass}
              onClick={() => pick(f.key)}
            >
              {f.label ?? f.key}
            </button>
          ))}
      </PopoverContent>
    </Popover>
  );
}
```

```ts
// apps/web/src/lib/safe-image-url.ts
// THREAT MODEL (Phase 1.8, mitigation 1+3): the ONLY gate for rendering a
// user-supplied cover URL. Consume its result ONLY as an <img src> — never in
// style/background-image/href/template strings. https-only by default.
export function safeImageUrl(value: unknown): string | null {
  if (typeof value !== 'string') return null;
  const trimmed = value.trim();
  if (trimmed === '') return null;
  let url: URL;
  try {
    url = new URL(trimmed);
  } catch {
    return null;
  }
  if (url.protocol !== 'https:') return null;
  return trimmed;
}
```

```tsx
      )}
```
