## What this builds

Two new saved-view types on Folio's existing list/kanban/table scaffolding:

1. **Time-based view** — positions work items on a date axis by a **user-selectable date field** (default `due_date`, mirroring how kanban lets you pick a group-by field). Auto-renders a span when both `start_date`+`end_date` are present, else a point. Unscheduled tray + horizontal drag-to-reschedule + group-by lanes reusing the kanban mechanism.
2. **Gallery view** — responsive card grid with an optional cover image from a `cover_url` frontmatter field (URL-based, https-only — **no upload, no asset storage**).

Plus a **view-type registry** refactor (Task 1) that makes adding the 3rd/4th view type one map entry instead of bespoke route wiring.

## Why these two, why now

User (Stefan) asked to scope time-based + a gallery view and explicitly asked whether the **date field should be selectable like kanban's group-by**. Answer: yes — the architecture already wants it (kanban stores a `groupBy` key + popover; the timeline parallels it with a `dateField` key + popover). That's the crux of the design and it's baked into the plan (Tasks 2 + 4).

## Threat model (embedded in the plan)

The gallery renders a **user-supplied URL** (`cover_url`) as an `<img src>` in *other members'* browsers — stored-injection / tracking-pixel surface (server never fetches it, so no SSRF). 5 numbered mitigations, the load-bearing ones:
1. One shared `safeImageUrl` (https-only, rejects `javascript:`/`data:`/etc., null-on-reject) — the single render gate.
2. `loading="lazy"` + `referrerPolicy="no-referrer"` on the cover `<img>`.
3. `cover_url` consumed ONLY via `safeImageUrl`, ONLY as `<img src>` — never `style`/`background`/`href`/template strings.
4. `dateField` + `type` Zod-validated at the views route.
5. No new fetch in either view (both use `useDocuments`) — also satisfies invariant 6.

`/code-review` at Task 9 verifies against these by number (verbatim invocation string is in the plan's Task 9 / the threat-model "how to use" closer).

## Gotchas the next session must respect (from project memory)

- **Migration journal:** every new `.sql` MUST get an entry in `apps/server/src/db/migrations/meta/_journal.json` — `migrate()` silently skips files not in the journal, and migration *tests bypass it* (false-positive). Task 2 step 4 covers this; Task 9 step 4 does a real `bun run db:migrate` to prove it.
- **Server tests run from INSIDE `apps/server`** (`cd apps/server && bun test`) — `bun test apps/server` from root fakes ~650 module-init failures.
- **Web tests use vitest** (`cd apps/web && npx vitest run`), NOT `bun test`. tsc is per-app (`bun x tsc --noEmit` from each of server/web/shared — no root tsconfig).
- **`list-view-create.test.tsx`** flakes under full-suite concurrency — rerun once before treating as a regression.
- **Verify subagent test counts + claims yourself** — re-run the suite after a subagent reports; spot-check claimed file references against source.
